## 1. ネットワークルール［通信ブロック］

通信そのものを根元から遮断します。DNR上限を圧迫しないよう、正規表現の乱用を避け、可能な限りプレーンなホスト名指定を行うのが基本です。ルールの末尾に `$` を付け、カンマ `,` 区切りで修飾子［Modifiers：フィルタが適用される条件や通信の制御方法を指定するためのパラメータ記法］を指定します。

### 1.1 基本的な適用と除外
### `$document` ［ページドキュメント全体の適用・除外］
* **構文例**： `||example.com^$document` ［ブロック］ / `@@||example.com^$document` ［例外］
* **概要**： ブラウザが最初に読み込むメインHTML［ドキュメント］そのものに対するリクエストにマッチします。
* **解説**： サイトが根本的に破損した場合のデバッグや、信頼できる特定サイトのみを全解除［ホワイトリスト化］する際に多用できる非常に強力な機能です。一方で、その強力さゆえに誤用すると広告を素通ししてしまうリスクを伴います。AdGuardではシステムレベルでの介入となるため、実機での慎重な検証が求められます。

### `$subdocument` ［サブドキュメントの適用・除外］
* **構文例**： `||example.com/ad-frame.html^$subdocument` ［ブロック］ / `@@||example.com/ad-frame.html^$subdocument` ［例外］
* **概要**： メインドキュメント内に埋め込まれた `<iframe>` や `<frame>` 要素を介して読み込まれる通信にマッチします。
* **解説**： 広告用フレームやSNSウィジェットを外枠ごと遮断でき、個別のスクリプトを追う必要がないためルール数を節約できます。ただし、決済フォームやログイン用外部フレームなどを巻き込んでサイトを破損させる誤爆［False Positive：本来ブロックすべきではないサイトの正規リソースや機能を、誤って広告やトラッキングと判定して遮断し、サイトの表示や動作を破損させてしまう現象］リスクには注意が必要です。

### `@@` ［例外ルール / ホワイトリスト］
* **構文例**： `@@||example.com/ads.js$script`
* **概要**： 先頭に付与することで、指定した条件に合致する通信や要素のブロックを解除します。
* **解説**： GitHub等でのFalse PositiveのIssue［イシュー：GitHub等のバージョン管理プラットフォームにおいて、ソフトウェアのバグ報告、機能リクエスト、フィルタの誤爆対応や改修要件を追跡・管理するためのタスクおよび議論のスレッド］報告時に、「このルールを追加すれば直る」という回避策［Workaround：根本的なフィルタやスクリプトの不具合修正が完了するまでの間、一時的に問題やサイト破損を回避するための応急的な処置や解決策］を即座に提示する用途で活躍します。ただし、条件を緩くしすぎるとブロックすべき悪質なトラッカーまで通過させてしまうため、リソースタイプ修飾子と併用して影響範囲を絞り込むことが必須となります。

### 1.2 対象と適用範囲の限定
### `$app` ［アプリ単位の適用限定］※ OSレベルアプリ特化
* **構文例**： `||ads.example.com^$app=com.twitter.android`
* **概要**： システム全体を監視するAdGuard環境において、特定のアプリにのみルールを適用します。
* **解説**： 他のアプリの正常なバックグラウンド通信を一切妨げずに特定アプリの広告のみを制御できる反面、デスクトップ版のブラウザ拡張機能では完全に無視されるため、クロスプラットフォームでのルール共有には不向きです。

### `$domain=` / `$denyallow=` ［適用先の限定と除外］
* **構文例**： `||example.com/ads.js$domain=test.com|~sub.test.com`
* **構文例**： `||example.com^$denyallow=sub.example.com` ［サブドメインの除外］ / `*$denyallow=com|net` ［特定のTLD〈Top-Level Domain / トップレベルドメイン：「.com」「.net」「.jp」など、ドメイン名において一番右側に位置する最上位の階層名〉の除外］
* **概要**： `$domain` は指定ドメインでのみ有効にし、`~` で特定サブドメインを除外します。`$denyallow` は「指定ドメイン以外」のすべてで適用します。
* **解説**： 複雑なサイト群におけるFalse Positiveを極めて正確かつ柔軟にコントロールできます。一方で、サイト運営側でサブドメインの運用が変更された際、ルールのメンテナンス漏れによるサイト破損を引き起こしやすい点に留意が必要です。

### `$to` ［リクエスト送信先ドメイン・ホストの指定］
* **構文例**： `||cdn.example.net/ads.js$to=bad-site.com|~good-site.com`
* **概要**： リクエストの「送信先［宛先］ホスト名」を条件に指定して適用範囲を限定します［`$domain=` がリクエストの「発信元ページ」を指定するのに対する概念です］。
* **解説**： 共有CDN［Content Delivery Network：Webページ・画像・スクリプトなどの大容量コンテンツを、ユーザーの地理的な近くから高速・安定して配信するために世界中に分散配置されたサーバーネットワーク］や汎用APIエンドポイントから配信される広告・トラッキングスクリプトを、特定の悪質なホストへ向かう通信に限定して遮断し、正規利用されているサイト側では許可するといった精緻なコントロールが可能です。ただし、接続元［発信元］と接続先［宛先］の関係性を正確に把握して記述する必要があり、ルール設計の難易度が高くなります。

### `$third-party` ［サードパーティリクエストの制限］
* **構文例**： `||tracker.com^$third-party`
* **概要**： リクエスト送信先が、現在閲覧中のドメインと異なる場合のみ適用されます。
* **解説**： CDNやファーストパーティの必須スクリプトを保護しつつ、外部からのトラッキング通信のみを効率よく遮断できます。しかし、CNAMEクローキング［CNAME Cloaking：サードパーティのトラッキング用ドメインを、サイトのファーストパーティのサブドメイン名としてDNS上で別名設定し、アドブロッカーやブラウザの追跡防止機能をすり抜ける手法］等を用いてファーストパーティを偽装する最新のトラッカーに対しては効果を発揮しません。

### `$strict-first-party` / `$strict-third-party` ［厳格なファースト/サードパーティ判定］
* **構文例**： `||tracker.example.com^$strict-third-party`
* **概要**： uBlock Originの仕様に準拠した修飾子で、フレームコンテキストやサブドメインの階層関係を厳格に評価し、真に同一のエンティティ［運営主体］間での通信かを判定してルールを適用します。
* **解説**： 従来の `$third-party` では区別が困難であった、複雑なiframeのネスト構造やファーストパーティを装う高度なトラッキング網に対し、極めて精緻なブロックを実行できます。これにより、uAssets［uBlock Origin公式フィルタ］からのルール移植における互換性と精度が大幅に向上するメリットがあります。反面、サイト認証のシングルサインオン［SSO］や外部連携型決済モジュールにおいて通信が遮断されるFalse Positiveを引き起こすリスクが高いため、実装時は厳密な例外指定［`@@`］との併用が必須となります。

### 1.3 リソースタイプとマッチング条件
### `$all` ［すべてのリソースタイプへの適用］
* **構文例**： `||malware.example.com^$all`
* **概要**： ドキュメント、スクリプト、画像、ポップアップなど、通信の種類を問わずすべてのリクエストに対してブロックを強制適用します。
* **解説**： 修飾子なしのプレーンなルールでは通常対象外となるメインページ自体の読み込み `$document` や別窓展開 `$popup` を含め、対象ドメインへの通信を完全に遮断します。マルウェア配布やフィッシングなど、ドメイン自体が完全に悪意を持つサイトをホストレベルで封鎖する用途において極めて強力かつ最適です。反面、適用範囲が最大であるため、正規サイトやCDNでFalse Positiveを発生させると「サイトへのアクセス自体が不可能になる」「広範なリソースが欠落する」といった致命的な破壊を招くため、確実なブラックリスト以外での汎用的な使用は厳重に避けるべきです。

### `$script`, `$image`, `$xmlhttprequest` 等 ［リソース指定］
* **構文例**： `||example.com/api^$xmlhttprequest`
* **概要**： 通信の種類を限定してルールを適用します。
* **解説**： ブロック対象を絞り込むことで、サイト必須リソースの巻き込みによるFalse Positiveを確実に防ぐことができます。反面、配信側が通信手法を変更［例: `script` から `fetch` への移行等］した際、容易にルールをすり抜けられるようになります。

### `$websocket` ［WebSocket接続の限定遮断］
* **構文例**： `||example.com/live-track^$websocket`
* **概要**： WebSocket［Web Real-Time Communicationとは異なる、ブラウザとWebサーバー間で一度確立したTCP接続を維持し、低遅延で双方向のデータ送受信を継続して行う通信プロトコル］による `ws://` および `wss://` 通信に対してのみルールを適用します。
* **解説**： 近年のSPA［Single Page Application：Webページ全体の再読み込みを行わず、初期読み込みした単一のHTML上で、ユーザーの操作に応じて必要なデータのみを非同期通信で取得し、画面のDOMツリーを動的に書き換えるモダンなWebアプリケーション構成］において、通常のHTTP通信を回避し、リアルタイムで広告配信やトラッキングデータを継続的に流し込む攻撃手法が急増しています。これらをピンポイントで遮断することで、ページの通常ロードを妨げずに追跡を無力化できます。反面、カスタマーサポートのチャットウィジェットやライブ配信のリアルタイムコメント等、正規の双方向通信機能まで破損させるFalse Positiveリスクが高い点に留意が必要です。

### `$webrtc` ［WebRTC接続の限定遮断］
* **構文例**： `||tracker.example.com^$webrtc`
* **概要**： WebRTC［Web Real-Time Communication：ブラウザ間でプラグインを介さずに音声・動画・データのリアルタイム双方向通信やP2P接続を行うためのオープンな通信規格］を介した通信に対してのみルールを適用します。
* **解説**： STUN/TURNサーバー［P2P通信を確立するために端末の通信経路やグローバルIPアドレスを解決・中継するためのサーバープロトコル］を悪用してブラウザのプライベートIPアドレスを漏洩させる高度なフィンガープリンティング［Fingerprinting：Cookieを使用する代わりに、ブラウザのバージョン、画面解像度、インストールされたフォント、GPU情報などの微細な環境特徴を組み合わせてユーザーや端末を一意に識別・追跡する技術］や、P2Pベースのトラッキングネットワークへの接続を根本から阻止できます。ただし、ブラウザを利用したWeb会議システムやP2Pコンテンツ配信が一切機能しなくなる致命的な副作用を伴うため、ドメイン指定との併用による対象限定が不可欠です。

### `$font`, `$media`, `$stylesheet` ［細分化されたリソースタイプ制御］
* **構文例**： `||fonts.example.net^$font` / `||video.example.com/ads/^$media`
* **概要**： Webフォント［`$font`］、動画・音声ストリームなどのHTML5メディア要素［`$media`］、スタイルシートCSS［`$stylesheet`］に限定してブロックを適用または例外解除します。
* **解説**： サードパーティ製Webフォントの読み込みを介したトラッキングの阻止や、モバイル回線での大容量な自動再生動画広告による通信量・バッテリー消費の浪費を確実に防止できます。一方で、 `$stylesheet` や `$font` を誤爆させると、サイトのアイコン［フォントアイコン］が消失したり、レイアウトが完全に崩壊するといった視覚的な問題が即座に発生します。

### `$method` ［HTTPリクエストメソッドの指定］
* **構文例**： `||example.com/api/track^$method=post|options` / `@@||example.com/api/login^$method=post`
* **概要**： GET、POST、OPTIONS、PUTなど、特定のHTTPリクエストメソッドに対してのみルールを適用または解除します。
* **解説**： 近年のトラッキング通信やアンチアドブロックはPOSTやOPTIONS［CORSプリフライトリクエスト：異なるドメイン間の通信において、実際のデータを送信する前に、サーバー側がその通信を許可しているかを確認するために自動送信されるOPTIONSメソッドの問い合わせリクエスト］を多用する傾向にあります。これらをピンポイントで遮断することで、通常のページロード［GETリクエスト］や正規APIを巻き込まずにトラッキングのみを無力化できる高い技術的精度を実現できます。反面、サイト仕様の変更によってメソッドが変更されると容易にすり抜けられる可能性があります。

### `$ping` ［トラッキング用ビーコン等の指定］
* **構文例**： `||example.com/track^$ping`
* **概要**： `navigator.sendBeacon()` や `<a ping>` 等によるPing/ビーコンリクエストに限定します。
* **解説**： ページ離脱時などにバックグラウンドで送信されるユーザーの行動トラッキング通信を確実に遮断できる機能ですが、サイト独自の計測APIなど、通常のXHR / Fetch通信で送られるトラッキングに対しては無効となります。

### `$popup` ［ポップアップの遮断］
* **構文例**： `||example.com^$popup`
* **概要**： 対象URLが新しいタブやウィンドウとして開かれる動作を検知してブロックします。
* **解説**： ユーザーの意図しない迷惑なリダイレクトやポップアップ広告を確実に防ぐことができる一方、正規の別窓リンク［ログイン画面や外部決済など］まで巻き込んで遮断してしまうFalse Positiveリスクが常に伴います。

### `$match-case` ［大文字・小文字の厳密な区別］
* **構文例**： `||example.com/AdBanner^$match-case`
* **概要**： URL文字列の大文字と小文字を厳密に区別して照合します。
* **解説**： 大 / 小文字が意味を持つ環境で正規リソースを巻き込まずに広告のみをピンポイントで遮断できます。ただし、配信側がURLを小文字に変更するだけで容易にすり抜けられるため、ルールの保守性が低下しやすい傾向があります。

### 1.4 通信の書き換えと動作変更
### `$empty` ［空のレスポンスの返却］
* **構文例**： `||example.com/tracker.js$empty`
* **概要**： 接続エラーではなくダミーの「空のレスポンス」を返却します。
* **解説**： スクリプトエラーを検知するアンチアドブロックを回避し、サイトの挙動崩れを防ぐことができます。その反面、通信自体は「200 OK」等として成功した扱いになるため、ブラウザ開発者ツールのネットワークパネルでのデバッグ時に直感的に把握しづらくなります。

### `$redirect` / `$redirect-rule` ［ローカルリソースへの置換］
* **構文例**： `||example.com/analytics.js$redirect=noopjs`
* **概要**： 外部スクリプトへのリクエストを遮断し、AdGuard内包のダミーリソースにすり替えます。
* **解説**： スクリプト読み込みが必須要件のサイトにおける「無限ロード」や「エラーによるレンダリング停止」を回避できます。一方で、MV3環境下など拡張機能がリダイレクトできるリソースに厳密な制限がある環境では、意図通りに動作しない場合があります。

### `$cookie` ［Cookieの制御・削除］
* **構文例**： `||example.com^$cookie=/.+/` ［対象ドメインの全Cookieを抑制］ / `@@||example.com^$cookie=/^session_/` ［特定Cookieの例外］
* **概要**： 通信自体は遮断せず、対象ドメインでやり取りされるHTTPヘッダーのCookieおよびSet-Cookieを抑制・削除します。
* **解説**： トラッキング用Cookieの付与を防ぐプライバシー保護のほか、訪問回数ベースで課金・ブロックを行うペイウォールの回避や、アンチアドブロック回避において通信を正常［200 OK］に見せかけつつ追跡のみを無効化する極めて高度で強力な手法です。反面、セッション維持やログイン認証に必要なCookieまで削除するとサイトへのログインや設定保存が一切できなくなる致命的な副作用があるため、正規表現や例外指定による厳格な対象限定が必須です。

### `$header` / `$removeheader` ［HTTPヘッダーの操作と削除］
* **構文例**： `||example.com^$removeheader=referer` / `||example.com^$header=permissions-policy: interest-cohort=()`
* **概要**： リクエスト/レスポンスから特定HTTPヘッダーを除去［`$removeheader`］、または任意のレスポンスヘッダーを上書き・注入［`$header`］します。
* **解説**： リファラ［アクセス元リンク］やUser-Agentを隠蔽して高いプライバシー保護とフィンガープリンティング耐性を実現したり、FLoC / Topics API［プライバシーサンドボックスにおけるブラウザベースの行動関心トラッキング技術：個人の詳細な閲覧履歴を外部に送信する代わりに、ブラウザ自身がユーザーの興味・関心トピックを算出し、広告配信業者に共有する仕組み］等のブラウザレベルトラッキングをヘッダーから剥奪できます。一方で、リファラやUAを厳密に検証する正規のセキュリティゲートウェイや決済システムにおいてアクセス拒否エラーを誘発するリスクや、CoreLibs［Android / Windowsアプリ版等］に深く依存し一部ブラウザ拡張機能では動作しないという環境依存性があります。

### `$urltransform` ［リクエストURLの動的書き換え］
* **構文例**： `||example.com/redirect?url=*$urltransform=/.*url=(.*)/$1/`
* **概要**： 正規表現とキャプチャグループを用いて、リクエストURLを送信前に別のURLへ動的に変換・リダイレクトさせます。
* **解説**： トラッキング用の中継サーバーをバイパスして目的のページへ直接遷移させたり、余計な追跡パラメータをURL構造ごと改変したりできます。しかし、正規表現の誤りによって無関係なURLが破損するリスクや、対応する環境［CoreLibs系アプリや特定のMV2 / MV3互換層］に留意が必要です。

### `$removeparam` ［トラッキングパラメータの高度な削除］
* **構文例**： `||example.com^$removeparam=/^utm_[a-z]+=/`
* **概要**： 正規表現を用いて特定のクエリパラメータを動的に削除します。
* **解説**： 通信自体は維持されるため、ページ遷移やリダイレクトを破壊せずにプライバシー保護を透過的に実行できる機能を持ちます。しかし、サイトのルーティングやセッション維持に必須のパラメータまで巻き込んで削除してしまうリスクがあります。

### `$replace` ［レスポンスボディの直接書き換え］
* **構文例**： `||example.com/api/$replace=/ad_enabled":true/ad_enabled":false/`
* **概要**： 正規表現を用いて、テキストベースのレスポンス中身を通信経路上で動的に置換します。
* **解説**： DOM操作の前に変数を書き換えられるため、極めて強力かつサイト側に検知されにくい特性があります。一方で、AdGuard for Windows / Android等のシステムレベル［MITM：Man-in-the-Middle / 中間者攻撃の技術を応用したローカルプロキシ：OSやシステムレベルで通信の間に入り込み、ローカルで生成した独自ルート証明書を用いてHTTPS暗号化通信を復号・検査・改変する仕組み］でのみ動作し、ブラウザ拡張機能では利用できない強い環境依存性を持ちます。

### `$csp` / `$permissions` ［セキュリティポリシーやブラウザ権限の強制制御］
* **構文例**： `||example.com^$csp=script-src 'self'` / `||example.com^$permissions=gyroscope=()`
* **概要**： HTTPレスポンスヘッダーにCSP［Content-Security-Policy / コンテンツセキュリティポリシー：クロスサイトスクリプティングやデータインジェクション等の攻撃を防ぐため、ブラウザが読み込み・実行を許可するリソースやドメインを厳格に制限するHTTPヘッダー］やPermissions-Policyを強制注入・変更します。
* **解説**： DOMに到達する前に悪意あるスクリプトを封じ込めたり、ジャイロセンサーや位置情報等のブラウザAPIへのアクセス権限を剥奪できます。ただし、サイト本来のCSPやポリシーと競合して正規スクリプトや機能が不全に陥るリスクが高く、一般ユーザーには原因特定が困難になりがちです。

### 1.5 デバッグ・ルールの優先度制御
### `$important` ［絶対的な優先適用］
* **構文例**： `||example.com/ads.js$important`
* **概要**： 例外ルールを含む、他のすべてのルールよりも優先してブロックを強制適用します。
* **解説**： 汎用例外ルールをすり抜けてくる悪質なトラッカーや広告を確実に遮断できる強力な手段となりますが、ルールの優先度階層を破壊するため、乱用すると意図しないブロックが発生した際のデバッグが極めて困難になります。

### `@@...$important` ［絶対的な例外・ホワイトリスト化］
* **構文例**： `@@||example.com/ads.js$important`
* **概要**： いかなる強力な遮断ルール［`$important` 付与を含むルール］の干渉も無効化し、最優先で通信を許可します。
* **解説**： サイト根幹を成す必須API等がサードパーティ汎用ルールによってFalse Positiveとなり遮断されている場合、確実に救済・復旧できます。しかし、ルールの権力が強すぎるため、指定が甘いと本来防ぐべき悪質トラッカーまで無条件に通過させてしまう重大なリスクを伴います。

### `$badfilter` ［既存ネットワークルールの無効化］
* **構文例**： `||example.com/ads.js$badfilter`
* **概要**： 購読しているフィルタに含まれる特定のネットワークルールを無効化します。
* **解説**： False Positiveの原因となっているサードパーティルールをピンポイントで解除でき、不具合のWorkaroundとして直結します。ただし、大元のルールが更新・削除された場合、このルール自体が不要なゴミとして残ってしまうため定期的な見直しが必要です。

### `$generichide` / `$genericblock` / `$specifichide` ［汎用および個別ルールの無効化］
* **構文例**： `@@||example.com^$generichide` ［汎用要素非表示の無効化］ / `@@||example.com^$specifichide` ［サイト固有の要素非表示のみを無効化］
* **概要**： 対象サイト上で、特定ドメインに縛られない汎用コスメティックルールや、逆にそのドメイン専用に書かれた固有のコスメティックルールを切り分けて無効化します。
* **解説**： サイト表示崩れが発生した際、「汎用ルールが原因か、そのサイト専用に書かれた個別ルールが原因か」を明確に切り分けるためのデバッグで極めて強力です。また、False Positiveを起こしている個別ルールを一時的に無効化しつつ基本的な広告除去機能［汎用ルール］を維持できるため、不具合対応のWorkaroundとしても最適です。

### `$stealth` ［ステルスモードの無効化］
* **構文例**： `@@||example.com^$stealth`
* **概要**： AdGuard独自のトラッキング保護機能による高度な介入［Cookieブロック等］を特定サイトで無効化します。
* **解説**： ログイン不可などの不具合が「通信ブロック」か「プライバシー保護機能の過剰介入」かを明確に見極めるデバッグに役立ちますが、対象サイトにおいてAdGuardの強力なプライバシー保護の恩恵が受けられなくなります。

## 2. コスメティックルール［DOM要素の非表示］

AdGuard for Androidでも軽く動作させるため、可能な限りシンプルな標準CSSセレクタ `##` を最優先します。

### 2.1 基本的な非表示・除外
### `##` ［基本のCSS非表示］
* **構文例**： `example.com##.ad-container`
* **概要**： ブラウザネイティブのレンダリングエンジンで直接処理される基本的な非表示記法です。
* **解説**： DOM操作の中で最も処理遅延とバッテリー消費が少ないため、モバイル環境において最優先で利用すべき記法ですが、クラス名やIDがランダム生成されるモダンな難読化広告要素の特定には無力です。

### `~` ［特定のドメインや要素の適用除外］
* **構文例**： `example.com,~shop.example.com##.ad-banner`
* **概要**： 非表示ルールの適用対象から特定のサブドメインやクラスを除外します。
* **解説**： 親ドメイン全体にルールを適用しつつ、表示崩れが起きる特定のサブドメインのみを容易にホワイトリスト化できますが、除外対象が長くなると可読性が下がり、ルール管理が複雑化します。

### 2.2 拡張セレクタによる高度な要素指定
### `#?#` ［拡張CSSルールの明示的適用］
* **構文例**： `example.com#?#.ad-box:contains(スポンサー)`
* **概要**： 標準CSSで指定できない要素を、AdGuard独自の拡張セレクタで指定します。
* **解説**： `:contains` 等を用いた複雑な要素指定ができ、構造が難読化された広告枠を特定できる強力な機能ですが、JavaScriptを介してDOMツリーを評価するため、多用するとページのレンダリング遅延やバッテリー消費を増大させます。

### `:is()` / `:where()` ［CSSセレクタのグルーピングと疑似クラス最適化］
* **構文例**： `example.com##:is(.ad, .banner):where([id*="sponsor"])`
* **概要**： 複数のセレクタを簡潔にまとめつつ、`:where()` は特異度［Specificity：CSSセレクタが特定の要素をどれだけ詳細・具体的に指示しているかを表す計算値で、複数のスタイルルールが競合した際にどの定義を最優先で適用するかを決定する基準］を0にリセットしてスタイル評価を行います。
* **解説**： AdGuard / uBlock Originの両環境および最新ブラウザでネイティブサポートされており、複雑なルール群のコード量を削減しつつスタイル計算のパフォーマンス・処理遅延を極小化できます。モバイル環境でのバッテリー消費低減に直結するベストプラクティスです。

### `:has` ［モダン疑似クラスの活用］
* **構文例**： `example.com##.article-card:has(> .sponsored-label)`
* **概要**： 特定の子要素を持つ親要素をターゲットにします。
* **解説**： 最新AdGuardエンジンではブラウザのネイティブCSS実装に寄せて処理負荷を抑えつつ複雑な構造を非表示にできます。一方で、ネイティブ非対応の古いブラウザ環境ではJSによるフォールバック評価が行われ、パフォーマンスが著しく低下します。

### `:contains` ［テキストフックによる要素指定］
* **構文例**： `example.com#?#.sponsor:contains(PR)`
* **概要**： 指定文字列を含む要素を非表示にします［uBlock Originの `:has-text` のAdGuard版］。
* **解説**： 構造が難読化された広告枠でも、視覚的なテキスト情報を手がかりに確実に消去できます。しかし、DOMツリー全体のテキストノードを走査するため、巨大なページではスクロール時のレンダリングジャンク［Rendering Junk：重いJavaScript処理や複雑なDOMツリー走査によってブラウザの処理負荷が高まり、画面の描画フレームレートが低下してスクロールやアニメーションがカクつく現象］の原因になり得ます。

### `:upward()` ［先祖要素への遡り］
* **構文例**： `example.com#?#.ad-text:upward(.ad-container)`
* **概要**： 引数に指定したCSSセレクタ、または数値分だけ階層を遡った先祖要素を探して非表示にします。
* **解説**： 広告テキスト等を起点にして、ランダムなIDを持つ親コンテナ全体をターゲットにする際に極めて有効な反面、過度な階層遡上はDOMの評価コストを上げ、False Positiveが発生した際の影響範囲が広くなります。

### `:xpath()` ［XPathを用いた要素指定］
* **構文例**： `example.com#?#:xpath(//div[@id="bar"]/p)`
* **概要**： XPath式を用いて対象要素を探索します。
* **解説**： 兄弟要素の配置関係など、CSSセレクタでは表現できない複雑な条件を持つ要素を指定できる最終手段として機能しますが、DOM評価の計算コストが非常に高いため、他の記法で代替できない場合のみに限定すべきです。

### `#$?#` ［正規表現による動的クラス対応］
* **構文例**： `example.com#$?#div[class=/^Ad__[a-zA-Z0-9]+$/]`
* **概要**： セレクタそのものに正規表現による柔軟なマッチングを適用します。
* **解説**： 頻繁に文字列が変化するスクリプト生成型の難読化広告枠に柔軟に追従できますが、スタイル再計算の負荷が極めて高く、バッテリー消費に直結するため乱用は厳禁です。

### `:matches-css()` / `:matches-css-before()` / `:matches-css-after()` ［計算済みスタイルや疑似要素に基づく非表示］
* **構文例**： `example.com#?#div:matches-css(position: fixed)` / `example.com#?#.banner:matches-css-before(content: *広告*)`
* **概要**： ブラウザが画面描画時に計算した実際のCSSプロパティ値や、`::before` および `::after` などの疑似要素に適用されたスタイル・テキスト内容を条件として要素を特定します。
* **解説**： クラス名やIDが完全に無意味なランダム文字列に生成され、かつテキストが疑似要素で描画される極めて高度な難読化広告や、画面を強制的に覆うアンチアドブロックオーバーレイを物理的な配置情報から正確に捕捉できる強力なセレクタです。しかし、ブラウザの計算済みスタイル［Computed Style］をリアルタイムで走査するため、DOM評価およびスタイル再計算のコストが全セレクタ中で最も高く、スマホ等では極端なレンダリングジャンクやバッテリー消費を招くため、他手法で解決できない場合のみの最終手段とすべきです。

### `:matches-attr()` ［属性値の正規表現マッチング］
* **構文例**： `example.com#?#div:matches-attr(data-ad-info=/^\[.*"sponsored".*\]$/)`
* **概要**： 特定のHTML属性値に対して正規表現を適用し、合致する要素を探索・特定します。
* **解説**： JSON文字列や複雑なパラメータが格納されたカスタムデータ属性［`data-*`］から、広告識別フラグのみを抽出してターゲティングできます。属性値内部を厳密に照合できるため高いブロック精度を誇りますが、DOMツリー内の全要素の属性文字列を解析するため、適切な親セレクタと組み合わせて走査範囲を限定しなければ、パフォーマンス遅延を発生させる要因になります。

### 2.3 要素に対するアクション［適用・削除］
### `:style(...)` ［CSSプロパティの安全な上書き］
* **構文例**： `example.com##body:style(overflow: auto !important;)`
* **概要**： 要素を隠すのではなく、指定したCSSプロパティを要素に直接適用します。
* **解説**： 強制注入 `#$#` より局所的に動作するため、スクロールロックの解除などを安全に行え、副作用を抑えられる特性があります。反面、特定の要素にしか影響を与えられないため、ページ全体に関わる複雑な動的スタイル変更には不向きです。

### `:remove()` ［DOMからの物理削除］
* **構文例**： `example.com##.floating-video:remove()`
* **概要**： `display: none` と異なり、DOMツリーから要素を完全に消去します。
* **解説**： 動画広告などのバックグラウンドでのメモリ消費とCPUリソースの浪費を根本から断ち切ることができますが、SPAでFalse Positiveを発生させると、ページをリロードしない限り復元できずサイト全体が機能不全に陥ります。

## 3. 高度なルール［スクリプトレット・HTML処理］

標準機能では突破できないアンチアドブロックに対する最終手段です。uBlock Origin特有の記法 `##+js` や `##^` 等は、近年のAdGuardの互換層により一部解釈・変換されるようになっていますが、完全な動作とパフォーマンス最適化を保証するためにはAdGuard固有の構文 `#%#//scriptlet` 等を使用することが強く推奨されます。

### 3.1 スクリプトレットの注入と例外
### `#%#//scriptlet` ［スクリプトレットによるJS関数ジャック］
* **構文例**： `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**： ページ読み込みの最初期にAdGuardの組み込みスクリプトを注入し、変数の固定化等を行います。
* **解説**： ネットワークブロック等では回避不可能な高度なJSベースのアンチアドブロックや強制ポップアップを完全に無力化できます。その反面、対象サイトの僅かなコード変更で即座にルールが陳腐化する欠点や、MV3環境での制限により将来的に機能しなくなるリスクがあります。

### `#%#//scriptlet('json-prune', ...)` / `json-prune` ［JSONデータの動的クレンジング］
* **構文例**： `example.com#%#//scriptlet('json-prune', 'ad_tags *.sponsor', 'ads_enabled')`
* **概要**： `fetch()` や `XMLHttpRequest` を介して受信されるJSONレスポンスをインターセプトし、指定したプロパティや広告データをパース直後に削除・改変します。
* **解説**： 動画プラットフォームやモダンなSPA、アプリ内通信において、データ構造の中に組み込まれた広告情報だけを外科手術のように正確に除去できます。ただし、対象となるJSONスキーマ［JSON Schema：JSONフォーマットのデータ構造において、キーの名前、データの型、階層構造などの必須構文ルールや構造定義を記述した仕様書］が変わると機能しなくなるため、サイト側の更新頻度が高い場合は定期的なメンテナンスが必要です。

### `#%#//scriptlet('prevent-fetch', ...)` / `#%#//scriptlet('prevent-xhr', ...)` ［非同期通信のJSレイヤー遮断］
* **構文例**： `example.com#%#//scriptlet('prevent-fetch', 'log_event')` / `example.com#%#//scriptlet('prevent-xhr', 'analytics')`
* **概要**： 特定の文字列や正規表現に合致する `fetch` または `XHR` リクエストの発行をJSの実行レイヤーで直接ブロックします。
* **解説**： ネットワークルールでのブロックを検知して無限ループやエラーメッセージを発生させるアンチアドブロックに対して、APIそのものをスタブ［Stub：元の複雑な広告処理スクリプトやトラッキング関数の代わりに、エラーを発生させずに処理を即座に終了させる無害な空の関数や固定値を返す仮のコード］化することで、エラーを発生させずにトラッキングや広告通信のみを無効化できます。

### `#%#//scriptlet('prevent-addEventListener', ...)` ［イベントリスナーの登録阻止］
* **構文例**： `example.com#%#//scriptlet('prevent-addEventListener', 'click', 'ad_tracker')`
* **概要**： `addEventListener` を介した特定のイベント［`click`、`scroll`、`copy` 等］へのイベントリスナー登録を、指定文字列に合致する関数に限定して初期段階でブロックします。
* **解説**： ユーザーの画面クリックやスクロールをトリガーとして発動するトラッキングスクリプトや、右クリック・コピー禁止・デバッガ検知といったAnnoyances［Annoyances：広告とは異なるものの、ユーザーのWebブラウジング体験を妨げるCookie同意バナー、ニュースレター登録依頼、右クリックやコピー禁止等の迷惑機能の総称］をピンポイントで無力化できます。反面、サイトの正規のクリック操作やスクロール連動UIの処理関数を誤って巻き込んでブロックすると、リンク遷移や画面スクロール自体が不可能になる重篤なFalse Positiveを招くリスクがあります。

### `#%#//scriptlet('set-local-storage-item', ...)` / `#%#//scriptlet('set-cookie', ...)` ［ストレージとCookieの強制設定・固定］
* **構文例**： `example.com#%#//scriptlet('set-local-storage-item', 'adblock_warning_shown', 'true')`
* **概要**： ブラウザの `localStorage` や `sessionStorage`、あるいは `document.cookie` に任意のキーと値を強制的に書き込み、サイトスクリプトによる上書きを阻止します。
* **解説**： CMP［Consent Management Platform：ユーザーに対して個人データ収集やCookie使用への同意を求めるプライバシーポップアップバナーを表示・管理するツールおよびプラットフォーム］による同意バナーの強制表示フラグや、アンチアドブロックの警告表示完了フラグを「表示済・同意済［ただしデータ収集は拒否］」の状態に偽装できます。通信やDOMを破壊せずに迷惑バナーや制限を無効化できるため極めて安全で処理負荷も軽い反面、サイトがストレージのキー名を変更した際には即座に再調査とルール更新が必要になります。

### `#%#//scriptlet('abort-on-property-read', ...)` / `#%#//scriptlet('abort-on-property-write', ...)` ［プロパティアクセス時のスクリプト強制停止］
* **構文例**： `example.com#%#//scriptlet('abort-on-property-read', 'window.adblockDetector')`
* **概要**： サイトスクリプトが特定のJSプロパティを読み込もうと［または書き込もうと］した瞬間、そのアクセスを捕捉してスタックトレースからスクリプトの実行そのものを即座に中断させます。
* **解説**： 難読化され、かつ動的に変数名やクラス名を変える強力なアンチアドブロック検知スクリプトであっても、アクセス先となるブラウザやグローバルの特定プロパティをフックすることで、検知ロジックの発動を完全に阻止できます。一方で、強制的にスクリプトの実行例外を発生させて止める仕様上、同じJSファイル内にある正常な描画処理や通信関数まで同伴して停止してしまうケースが多く、サイト破損のデバッグと影響範囲の特定に高度な技術知識を要します。

### `#%#//scriptlet('remove-attr', ...)` / `#%#//scriptlet('remove-class', ...)` ［属性・クラスの動的削除］
* **構文例**： `example.com#%#//scriptlet('remove-attr', 'target', 'a[href*="ad.example"]')`
* **概要**： 指定したDOM要素から特定の属性やクラス名を動的に削除します。
* **解説**： クリック時に別窓を開かせる `target="_blank"` や、スクロールをロックする `class="modal-open"`、遅延読み込みで広告を発動させるカスタム属性などを安全に解除できます。CSSや `:remove()` では対応できない動的な挙動の修正に最適です。

### `#@%#` / `#@?#` ［拡張ルールのピンポイント例外］
* **構文例**： `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**： 当該スクリプトレットの注入などを局所的に無効化し、動作を復旧させます。
* **解説**： 購読フィルタが原因でサイトの正規の処理が止まってしまった場合のWorkaroundとして効果的な解決策となりますが、どのスクリプトレットが原因で破損しているかを特定するためには、高度なデバッグスキルが要求されます。

### 3.2 HTML・CSSの強制操作
### `$$` ［HTMLフィルタリングルール］
* **構文例**： `example.com$$script[tag-content="ad-script-content"]`
* **概要**： ブラウザがDOMツリーを構築する前の生HTML応答データの段階で、対象要素を直接物理的に削除します。
* **解説**： DOM読み込みと同時に実行されてしまう悪質なインラインスクリプト等を、パースされる前に根本から消去できる極めて強力な機能です。一方で、指定できるセレクタが限定的であり、適用を誤るとページ構造自体を完全に破壊してしまいます。

### `#$#` ［CSSの強制注入］
* **構文例**： `example.com#$#body { overflow: visible !important; }`
* **概要**： 任意のCSSプロパティをページ全体に直接注入します。
* **解説**： コンテンツブロッカーを検知してスクロールをロックする悪質なオーバーレイの制限などを強制解除できますが、サイト自体の正規のレイアウト更新と競合し、予期せぬ表示崩れ［Z-indexの破綻など］を招くリスクが常に伴います。

* **[独自の広告フィルタを作成する方法 | AdGuard Knowledge Base](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)**

  AdGuard特有の文法や推奨作法を体系的に習得できます。uBlock Originとの構文の違い `##+js(...)` ではなく `#%#//scriptlet(...)`、`:has-text(...)` ではなく `:contains(...)` を用いる点や、拡張CSSルール `#?#` 等の理解に役立ちます。一方、最新のアンチアドブロック突破などの実践的ロジックは網羅されていないため、uAssets等との併用が必要となります。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)**

  AdGuard公式の最新ルールリポジトリであり、モバイルアプリ向けの指定 `$app` や独自スクリプトレットなど、AdGuard環境に特化した実践的なルールの記述法を直接参照できる最大の強みがあります。ただし、ルール群は用途別［Base, Mobile, Annoyances等］にモジュール構造［Module Structure：大規模なフィルタ群やプログラムを、単一の巨大なファイルとして管理するのではなく、用途、プラットフォーム、機能ごとに独立した複数のファイルやディレクトリに分割・整理する設計手法］として細かくディレクトリ分割されているため、目的の記述を探し出すにはリポジトリの設計方針に対する深い理解が求められます。

* **[AdGuard Blog](https://adguard.com/en/blog/index.html)**

  広告ブロックの最新トレンドやMV3対応ロードマップ等を発信する公式ブログです。業界全体の動向や大型アップデートを追うのに有用ですが、一般向けのキャンペーン記事も多く混在しています。なお、疑似クラス `:has(...)` 等のネイティブ実装状況や、処理負荷を低減するフィルタ記述のベストプラクティスといった専門情報はここには掲載されないため、高度な開発情報は公式ドキュメントやGitHub［CoreLibs等］を別途参照する必要があります。

* **[easylist / easylist](https://github.com/easylist/easylist)**

  すべての原点である汎用ネットワークルールの集積であり、長年のコミュニティの運用によって高度に洗練された、最も堅牢で安定したルールの記述法を学べるリポジトリです。MV3環境や最新の難読化広告にはこれ単体では対応しきれない側面もあります。また、独自の検証環境でテストしたルール群やスクリプトをコミュニティ内で迅速に共有する際には、GitHub Gist［Gist：GitHubが提供する、単一または複数のコードスニペットや設定ファイル、カスタムルールリストなどを簡単に公開・管理・共有できる簡易リポジトリサービス］が活用されることも多くあります。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)**

  uBlockOrigin / uAssetsには高度な動的要素指定やアンチアドブロック突破のロジック群がまとめられており、現代最高峰のフィルタリング技術の参考として有用です。AdGuard環境へ適用する場合、現在では `##+js(...)` 構文がそのまま解釈されることが多いですが、uBlock Origin固有の特権スクリプトレット［trusted系など］の存在や引数の仕様差異があるため、AdGuard側のエンジンとの互換性確認や代替ロジックへの調整が必要となります。
