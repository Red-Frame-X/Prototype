# AdGuard Custom Rules Reference

MV3（Manifest V3）環境の厳格なDNR（Declarative Net Request）ルール上限や、AdGuard for Androidにおける処理遅延・バッテリー消費の極小化を前提とした、上級者向けのカスタムルール・リファレンスです。
通信遮断、DOM操作、スクリプト注入の各アプローチについて、AdGuard専用の記法 `#$#`、`#?#`、`:contains`、`#%#//scriptlet` 等を用いた最適な運用手法と注意点を整理しています。

## 1. ネットワークルール（通信ブロック）

通信そのものを根元から遮断します。DNR上限を圧迫しないよう、正規表現の乱用を避け、可能な限りプレーンなホスト名指定を行うのが基本です。ルールの末尾に `$` を付け、カンマ `,` 区切りで修飾子（Modifiers）を指定します。

### 1.1 基本的な適用と除外
### `$document` （ページドキュメント全体の適用・除外）
* **構文例**： `||example.com^$document` （ブロック） / `@@||example.com^$document` （例外）
* **概要**： ブラウザが最初に読み込むメインHTML（ドキュメント）そのものに対するリクエストにマッチします。
* **解説**： サイトが根本的に破損した場合のデバッグや、信頼できる特定サイトのみを全解除（ホワイトリスト化）する際に多用できる非常に強力な機能です。一方で、その強力さゆえに誤用すると広告を素通ししてしまうリスクを伴います。AdGuardではシステムレベルでの介入となるため、実機での慎重な検証が求められます。

### `$subdocument` （サブドキュメントの適用・除外）
* **構文例**： `||example.com/ad-frame.html^$subdocument`
* **概要**： メインドキュメント内に埋め込まれた `<iframe>` や `<frame>` 要素を介して読み込まれる通信にマッチします。
* **解説**： 広告用フレームやSNSウィジェットを外枠ごと遮断でき、個別のスクリプトを追う必要がないためルール数を節約できます。ただし、決済フォームやログイン用外部フレームなどを巻き込んでサイトを破損させる誤爆リスクには注意が必要です。

### `@@` （例外ルール / ホワイトリスト）
* **構文例**： `@@||example.com/ads.js$script`
* **概要**： 先頭に付与することで、指定した条件に合致する通信や要素のブロックを解除します。
* **解説**： サイト破損の不具合報告時に「このルールを追加すれば直る」という解決策（ワークアラウンド）を即座に提示する用途で活躍します。ただし、条件を緩くしすぎるとブロックすべき悪質なトラッカーまで通過させてしまうため、リソースタイプ修飾子と併用して影響範囲を絞り込むことが必須となります。

### 1.2 対象と適用範囲の限定
### `$app` （アプリ単位の適用限定）※ OSレベルアプリ特化
* **構文例**： `||ads.example.com^$app=com.twitter.android`
* **概要**： システム全体を監視するAdGuard環境において、特定のアプリにのみルールを適用します。
* **解説**： 他のアプリの正常なバックグラウンド通信を一切妨げずに特定アプリの広告のみを制御できる反面、デスクトップ版のブラウザ拡張機能では完全に無視されるため、クロスプラットフォームでのルール共有には不向きです。

### `$domain=` / `$denyallow=` （適用先の限定と除外）
* **構文例**： `||example.com/ads.js$domain=test.com|~sub.test.com`
* **概要**： `$domain` は指定ドメインでのみ有効にし、`~` で特定サブドメインを除外します。`$denyallow` は「指定ドメイン以外」のすべてで適用します。
* **解説**： 複雑なサイト群における誤爆（False Positive）を極めて正確かつ柔軟にコントロールできます。一方で、サイト運営側でサブドメインの運用が変更された際、ルールのメンテナンス漏れによるサイト破損を引き起こしやすい点に留意が必要です。

### `$third-party` （サードパーティリクエストの制限）
* **構文例**： `||tracker.com^$third-party`
* **概要**： リクエスト送信先が、現在閲覧中のドメインと異なる場合のみ適用されます。
* **解説**： CDNやファーストパーティの必須スクリプトを保護しつつ、外部からのトラッキング通信のみを効率よく遮断できます。しかし、CNAMEクローキング等を用いてファーストパーティを偽装する最新のトラッカーに対しては効果を発揮しません。

### 1.3 リソースタイプとマッチング条件
### `$script`, `$image`, `$xmlhttprequest` 等 （リソース指定）
* **構文例**： `||example.com/api/^$xmlhttprequest`
* **概要**： 通信の種類を限定してルールを適用します。
* **解説**： ブロック対象を絞り込むことで、サイト必須リソースの巻き込み誤爆を確実に防ぐことができます。反面、配信側が通信手法を変更（例: script から fetch への移行等）した際、容易にルールをすり抜けられるようになります。

### `$ping` / `$beacon` （トラッキング用ビーコンの指定）
* **構文例**： `||example.com/track^$ping`
* **概要**： `navigator.sendBeacon()` 等によるPing/ビーコンリクエストに限定します。
* **解説**： ページ離脱時などにバックグラウンドで送信されるユーザーの行動トラッキング通信を確実に遮断できる機能ですが、サイト独自の計測APIなど、通常のXHR/Fetch通信で送られるトラッキングに対しては無効となります。

### `$popup` （ポップアップの遮断）
* **構文例**： `||example.com^$popup`
* **概要**： 対象URLが新しいタブやウィンドウとして開かれる動作を検知してブロックします。
* **解説**： ユーザーの意図しない迷惑なリダイレクトやポップアップ広告を確実に防ぐことができる一方、正規の別窓リンク（ログイン画面や外部決済など）まで巻き込んで遮断してしまう誤爆リスクが常に伴います。

### `$match-case` （大文字・小文字の厳密な区別）
* **構文例**： `||example.com/AdBanner^$match-case`
* **概要**： URL文字列の大文字と小文字を厳密に区別して照合します。
* **解説**： 大/小文字が意味を持つ環境で正規リソースを巻き込まずに広告のみをピンポイントで遮断できます。ただし、配信側がURLを小文字に変更するだけで容易にすり抜けられるため、ルールの保守性が低下しやすい傾向があります。

### 1.4 通信の書き換えと動作変更
### `$empty` （空のレスポンスの返却）
* **構文例**： `||example.com/tracker.js$empty`
* **概要**： 接続エラーではなくダミーの「空のレスポンス」を返却します。
* **解説**： スクリプトエラーを検知するアンチアドブロックを回避し、サイトの挙動崩れを防ぐことができます。その反面、通信自体は「200 OK」等として成功した扱いになるため、ネットワークパネルでのデバッグ時に直感的に把握しづらくなります。

### `$redirect` / `$redirect-rule` （ローカルリソースへの置換）
* **構文例**： `||example.com/analytics.js$redirect=noopjs`
* **概要**： 外部スクリプトへのリクエストを遮断し、AdGuard内包のダミーリソースにすり替えます。
* **解説**： スクリプト読み込みが必須要件のサイトにおける「無限ロード」や「エラーによるレンダリング停止」を回避できます。一方で、MV3環境下など拡張機能がリダイレクトできるリソースに厳密な制限がある環境では、意図通りに動作しない場合があります。

### `$removeparam` （トラッキングパラメータの高度な削除）
* **構文例**： `||example.com^$removeparam=/^utm_[a-z]+=/`
* **概要**： 正規表現を用いて特定のクエリパラメータを動的に削除します。
* **解説**： 通信自体は維持されるため、ページ遷移やリダイレクトを破壊せずにプライバシー保護を透過的に実行できる機能を持ちます。しかし、サイトのルーティングやセッション維持に必須のパラメータまで巻き込んで削除してしまうリスクがあります。

### `$replace` （レスポンスボディの直接書き換え）
* **構文例**： `||example.com/api/$replace=/ad_enabled":true/ad_enabled":false/`
* **概要**： 正規表現を用いて、テキストベースのレスポンス中身を通信経路上で動的に置換します。
* **解説**： DOM操作の前に変数を書き換えられるため、極めて強力かつサイト側に検知されにくい特性があります。一方で、AdGuard for Windows/Android等のシステムレベル（MITM）でのみ動作し、ブラウザ拡張機能では利用できない強い環境依存性を持ちます。

### `$csp` （Content-Security-Policyの強制適用）
* **構文例**： `||example.com^$csp=script-src 'self'`
* **概要**： HTTPレスポンスヘッダーにCSPを強制注入します。
* **解説**： DOMに到達する前に悪意あるスクリプトをブラウザの根元から強力に封じ込めることができますが、サイト本来のCSPと競合して正規スクリプトまで機能不全に陥るリスクが高く、一般ユーザーには原因特定が困難になりがちです。

### 1.5 デバッグ・ルールの優先度制御
### `$important` （絶対的な優先適用）
* **構文例**： `||example.com/ads.js$important`
* **概要**： 例外ルールを含む、他のすべてのルールよりも優先してブロックを強制適用します。
* **解説**： 汎用例外ルールをすり抜けてくる悪質なトラッカーや広告を確実に遮断できる強力な手段となりますが、ルールの優先度階層を破壊するため、乱用すると意図しないブロックが発生した際のデバッグが極めて困難になります。

### `@@...$important` （絶対的な例外・ホワイトリスト化）
* **構文例**： `@@||example.com/ads.js$important`
* **概要**： いかなる強力な遮断ルール（`$important` 付与含む）の干渉も無効化し、最優先で通信を許可します。
* **解説**： サイト根幹を成す必須API等がサードパーティ汎用ルールによって誤爆・遮断されている場合、確実に救済・復旧できます。しかし、ルールの権力が強すぎるため、指定が甘いと本来防ぐべき悪質トラッカーまで無条件に通過させてしまう重大なリスクを伴います。

### `$badfilter` （既存ネットワークルールの無効化）
* **構文例**： `||example.com/ads.js$badfilter`
* **概要**： 購読しているベースフィルタに含まれる特定のネットワークルールを無効化します。
* **解説**： 誤爆の原因となっているサードパーティルールをピンポイントで解除でき、不具合修正に直結します。ただし、大元のルールが更新・削除された場合、このルール自体が不要なゴミとして残ってしまうため定期的な見直しが必要です。

### `$generichide` / `$genericblock` （汎用ルールの無効化）
* **構文例**： `@@||example.com^$generichide`
* **概要**： 対象サイト上で、特定ドメインに縛られない汎用コスメティックルールや汎用ネットワークルールをすべて無効化します。
* **解説**： サイト表示崩れの原因が「汎用ルールの巻き込み」か「特定サイト向けルールの誤爆」かを切り分けるデバッグ作業で極めて有効です。反面、適用すると無害な広告まで大量に再表示されるため、原因特定後のピンポイントな修正ルールの作成が前提となります。

### `$stealth` （ステルスモードの無効化）
* **構文例**： `@@||example.com^$stealth`
* **概要**： AdGuard独自のトラッキング保護機能による高度な介入（Cookieブロック等）を特定サイトで無効化します。
* **解説**： ログイン不可などの不具合が「通信ブロック」か「プライバシー保護機能の過剰介入」かを明確に見極めるデバッグに役立ちますが、対象サイトにおいてAdGuardの強力なプライバシー保護の恩恵が受けられなくなります。

## 2. コスメティックルール（DOM要素の非表示）

AdGuard for Androidでも軽く動作させるため、可能な限りシンプルな標準CSSセレクタ `##` を最優先します。

### 2.1 基本的な非表示・除外
### `##` （基本のCSS非表示）
* **構文例**： `example.com##.ad-container`
* **概要**： ブラウザネイティブのレンダリングエンジンで直接処理される基本的な非表示記法です。
* **解説**： DOM操作の中で最も処理遅延とバッテリー消費が少ないため、モバイル環境において最優先で利用すべき記法ですが、クラス名やIDがランダム生成されるモダンな難読化広告要素の特定には無力です。

### `~` （特定のドメインや要素の適用除外）
* **構文例**： `example.com,~shop.example.com##.ad-banner`
* **概要**： 非表示ルールの適用対象から特定のサブドメインやクラスを除外します。
* **解説**： 親ドメイン全体にルールを適用しつつ、表示崩れが起きる特定のサブドメインのみを容易にホワイトリスト化できますが、除外対象が長くなると可読性が下がり、ルール管理が複雑化します。

### 2.2 拡張セレクタによる高度な要素指定
### `#?#` （拡張CSSルールの明示的適用）
* **構文例**： `example.com#?#.ad-box:contains(スポンサー)`
* **概要**： 標準CSSで指定できない要素を、AdGuard独自の拡張セレクタで指定します。
* **解説**： `:contains` 等を用いた複雑な要素指定ができ、構造が難読化された広告枠を特定できる強力な機能ですが、JavaScriptを介してDOMツリーを評価するため、多用するとページのレンダリング遅延やバッテリー消費を増大させます。

### `:has` （モダン疑似クラスの活用）
* **構文例**： `example.com##.article-card:has(> .sponsored-label)`
* **概要**： 特定の子要素を持つ親要素をターゲットにします。
* **解説**： 最新AdGuardエンジンではブラウザのネイティブCSS実装に寄せて処理負荷を抑えつつ複雑な構造を非表示にできます。一方で、ネイティブ非対応の古いブラウザ環境ではJSによるフォールバック評価が行われ、パフォーマンスが著しく低下します。

### `:contains` （テキストフックによる要素指定）
* **構文例**： `example.com#?#.sponsor:contains(PR)`
* **概要**： 指定文字列を含む要素を非表示にします（uBOの `:has-text` のAdGuard版）。
* **解説**： 構造が難読化された広告枠でも、視覚的なテキスト情報を手がかりに確実に消去できます。しかし、DOMツリー全体のテキストノードを走査するため、巨大なページではスクロール時の描画遅延（ジャンク）の原因になり得ます。

### `:upward()` （先祖要素への遡り）
* **構文例**： `example.com#?#.ad-text:upward(.ad-container)`
* **概要**： 引数に指定したCSSセレクター、または数値分だけ階層を遡った先祖要素を探して非表示にします。
* **解説**： 広告テキスト等を起点にして、ランダムなIDを持つ親コンテナ全体をターゲットにする際に極めて有効な反面、過度な階層遡上はDOMの評価コストを上げ、誤爆した際の影響範囲が広くなります。

### `:xpath()` （XPathを用いた要素指定）
* **構文例**： `example.com#?#:xpath(//div[@id="bar"]/p)`
* **概要**： XPath式を用いて対象要素を探索します。
* **解説**： 兄弟要素の配置関係など、CSSセレクターでは表現できない複雑な条件を持つ要素を指定できる最終手段として機能しますが、DOM評価の計算コストが非常に高いため、他の記法で代替できない場合のみに限定すべきです。

### `#$?#` （正規表現による動的クラス対応）
* **構文例**： `example.com#$?#div[class^="Ad__"]`
* **概要**： セレクタそのものに正規表現による柔軟なマッチングを適用します。
* **解説**： 頻繁に文字列が変化するスクリプト生成型の難読化広告枠に柔軟に追従できますが、スタイル再計算の負荷が極めて高く、バッテリー消費に直結するため乱用は厳禁です。

### 2.3 要素に対するアクション（適用・削除）
### `:style(...)` （CSSプロパティの安全な上書き）
* **構文例**： `example.com##body:style(overflow: auto !important;)`
* **概要**： 要素を隠すのではなく、指定したCSSプロパティを要素に直接適用します。
* **解説**： 強制注入 `#$#` より局所的に動作するため、スクロールロックの解除などを安全に行え、副作用を抑えられる特性があります。反面、特定の要素にしか影響を与えられないため、ページ全体に関わる複雑な動的スタイル変更には不向きです。

### `:remove()` （DOMからの物理削除）
* **構文例**： `example.com##.floating-video:remove()`
* **概要**： `display: none` と異なり、DOMツリーから要素を完全に消去します。
* **解説**： 動画広告などのバックグラウンドでのメモリ消費とCPUリソースの浪費を根本から断ち切ることができますが、SPA（Single Page Application）で誤爆すると、ページをリロードしない限り復元できずサイト全体が機能不全に陥ります。

## 3. 高度なルール（スクリプトレット・HTML処理）

標準機能では突破できないアンチアドブロックに対する最終手段です。uBlock Origin特有の記法（`##+js` や `##^` 等）は解釈されないため、AdGuard固有の構文を使用します。

### 3.1 スクリプトレットの注入と例外
### `#%#//scriptlet` （スクリプトレットによるJS関数ジャック）
* **構文例**： `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**： ページ読み込みの最初期にAdGuardの組み込みスクリプトを注入し、変数の固定化等を行います。
* **解説**： ネットワークブロック等では回避不可能な高度なJSベースのアンチアドブロックや強制ポップアップを完全に無力化できます。その反面、対象サイトの僅かなコード変更で即座にルールが陳腐化する欠点や、MV3環境での制限により将来的に機能しなくなるリスクがあります。

### `#@%#` / `#@?#` （拡張ルールのピンポイント例外）
* **構文例**： `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**： 当該スクリプトレットの注入などを局所的に無効化し、動作を復旧させます。
* **解説**： 購読フィルタが原因でサイトの正規の処理が止まってしまった場合の効果的な解決策となりますが、どのスクリプトレットが原因で破損しているかを特定するためには、高度なデバッグスキルが要求されます。

### 3.2 HTML・CSSの強制操作
### `$$` （HTMLフィルタリングルール）
* **構文例**： `example.com$$script[tag-content="ad-script-content"]`
* **概要**： ブラウザがDOMツリーを構築する前の生HTML応答データの段階で、対象要素を直接物理的に削除します。
* **解説**： DOM読み込みと同時に実行されてしまう悪質なインラインスクリプト等を、パースされる前に根本から消去できる極めて強力な機能です。一方で、指定できるセレクタが限定的であり、適用を誤るとページ構造自体を完全に破壊してしまいます。

### `#$#` （CSSの強制注入）
* **構文例**： `example.com#$#body { overflow: visible !important; }`
* **概要**： 任意のCSSプロパティをページ全体に直接注入します。
* **解説**： コンテンツブロッカーを検知してスクロールをロックする悪質なオーバーレイの制限などを強制解除できますが、サイト自体の正規のレイアウト更新と競合し、予期せぬ表示崩れ（Z-indexの破綻など）を招くリスクが常に伴います。

## 4. 参考・引用元ソース（最新動向とリポジトリ）

* **[独自の広告フィルタを作成する方法 | AdGuard Knowledge Base](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)**

  AdGuard特有の文法や推奨作法を体系的に習得できます。uBlock Originとの構文の違い `##+js(...)` ではなく `#%#//scriptlet(...)`、`:has-text(...)` ではなく `:contains(...)` を用いる点や、拡張CSSルール `#?#` 等の理解に役立ちます。一方、最新のアンチアドブロック突破などの実践的ロジックは網羅されていないため、uAssets等との併用が必要となります。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)**

  AdGuard公式の最新リポジトリであり、Android環境へのシステムレベル対応やMV3向け最適化など、AdGuard固有の実践的なコードを直接参照できる最大の強みがあります。ルール数が膨大であるため、目的の記述を探し出すにはリポジトリ構造の深い理解が求められます。

* **[AdGuard Blog](https://adguard.com/en/blog/index.html)**

  広告ブロックの最新トレンドやMV3対応ロードマップ等を発信する公式ブログです。疑似クラス `:has(...)` 等のネイティブ実装状況など、処理負荷を低減する最新のベストプラクティスをいち早く追える有用な情報源です。ただし、一般向けのキャンペーン記事も多く混在しており、フィルタ開発に直結する専門情報を抽出するにはスクリーニングの手間がかかります。

* **[easylist / easylist](https://github.com/easylist/easylist)**

  すべての原点である汎用ネットワークルールの集積であり、最も枯れた安定したルールの記述法を学べるリポジトリです。MV3環境や最新の難読化広告にはこれ単体では対応しきれない側面もあります。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)**

  高度な動的要素指定やアンチアドブロック突破のロジック群がまとめられており、現代最高峰のフィルタリング技術の参考として有用です。AdGuard環境へ適用するには構文変換として `##+js(...)` を `#%#//scriptlet(...)` に書き換える工程が必須となります。
