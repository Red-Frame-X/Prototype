# AdGuard Custom Rules Reference

MV3（Manifest V3）環境の厳格なDNR（Declarative Net Request）ルール上限や、AdGuard for Androidにおける処理遅延・バッテリー消費の極小化を前提とした、上級者向けのカスタムルール・リファレンスです。
通信遮断、DOM操作、スクリプト注入の各アプローチについて、AdGuard専用の記法 `#$#`、`#?#`、`:contains`、`#%#//scriptlet` 等を用いた最適な運用手法と注意点を整理しています。

## 1. ネットワークルール（通信ブロック）

通信そのものを根元から遮断します。DNR上限を圧迫しないよう、正規表現の乱用を避け、可能な限りプレーンなホスト名指定を行うのが上級者の基本です。ルールの末尾に `$` を付け、カンマ `,` 区切りで修飾子（Modifiers）を指定します。不具合報告時の原因切り分けにも頻繁に使用されます。

### 基本構文と適用範囲
### `$app` （アプリ単位の適用限定）※ OSレベルアプリ（Android/Windows/Mac等）特化
* **構文例**： `||ads.example.com^$app=com.twitter.android`
* **概要**： システム全体を監視するAdGuard for AndroidやWindows、Macなどの環境において、特定のアプリにのみルールを適用するための指定です。
* **解説**： 他のアプリの正常なバックグラウンド通信を妨げないという大きな利点がありますが、デスクトップ版のブラウザ拡張機能では完全に無視されるため、クロスプラットフォームでのルール共有には不向きです。

### `$document` （ページドキュメント全体の適用・除外）
* **構文例**： `||example.com^$document` （ブロック） / `@@||example.com^$document` （例外）
* **概要**： ブラウザが最初に読み込むメインHTML（ドキュメント）そのものに対するリクエストにマッチします。
* **ブロックの挙動**： 指定ドメインのページ全体に対してフィルタリングを強制適用します。
* **例外の挙動**： `@@` を付与することで、当該サイトにおけるフィルタリング機能（ネットワーク遮断および非表示ルール）をすべて無効化し、ホワイトリスト化します。
* **解説**： サイトが根本的に破損した場合のデバッグや、信頼できる特定のサイトのみ全解除する際に多用されます。ただし、非常に強力な設定であるため誤用には注意が必要です。uBlock Originと比較すると、システムレベルでの介入を行うAdGuardではフィルタリング処理の優先順位や介入範囲が異なるため、実機での検証が推奨されます。

### `$domain=` / `$denyallow=` （適用先の限定と除外）
* **構文例**： `||example.com/ads.js$domain=test.com|~sub.test.com`
* **概要**： `$domain` は指定ドメインでのみブロックを有効にし、`~` で特定サブドメインを除外します。`$denyallow` は逆に「指定したドメイン以外」のすべての場所でブロックを適用します。
* **解説**： 複雑なサイト群における誤爆（False Positive）を極めて正確にコントロールできる強力な機能ですが、サブドメインの運用が変更された際、ルールのメンテナンス漏れによるサイト破損を引き起こしやすい点に注意が必要です。

### `$subdocument` （サブドキュメントの適用・除外）
* **構文例**： `||example.com/ad-frame.html^$subdocument` （ブロック） / `@@||example.com/ad-frame.html^$subdocument` （例外）
* **概要**： メインドキュメント内に埋め込まれた `<iframe>` や `<frame>` 要素を介して読み込まれる「入れ子状の外部ページ」に対するリクエストにマッチします。
* **ブロックの挙動**： フレーム単位での読み込みそのものを遮断します。
* **例外の挙動**： `@@` を付与することで、特定のiframe内へのフィルタリングを個別に解除します。
* **解説**： 広告用フレームやSNSウィジェットなど、ページの一部として外部から読み込まれるコンテンツを外枠ごと遮断できます。個別のスクリプトを追う必要がなくルール数を節約できるメリットがある一方、決済フォームやログイン用フレームなどを巻き込んで破損させるリスクがあります。uBlock Originに比べ、システムレベルでプロキシ的に通信を処理するAdGuardでは、フレーム内のトラフィックに対する遮断のタイミングや範囲が厳密に評価される傾向があります。

### `@@` （例外ルール / ホワイトリスト）
* **構文例**： `@@||example.com/ads.js$script`
* **概要**： 先頭に付与することで、指定した条件に合致する通信や要素のブロックを解除（ホワイトリスト化）します。
* **解説**： サイト破損の不具合報告時に「このルールを追加すれば直る」という解決策（ワークアラウンド）を提示する際の基本構文です。しかし、条件を緩くしすぎるとブロックすべきトラッカーまで通してしまうため、リソースタイプ修飾子などと併用して影響範囲を最小限に留める必要があります。

### マッチング条件・リソースタイプの指定
### `$third-party` （サードパーティリクエストの制限）
* **構文例**： `||tracker.com^$third-party`
* **概要**： リクエストの送信先が、現在閲覧しているサイトのドメインと異なる（サードパーティである）場合のみ適用されます。
* **解説**： CDNやファーストパーティとして動作する必須スクリプトを保護しつつ、外部からのトラッキング通信のみを効率よく遮断できます。ただし、CNAMEクローキングなどを用いてファーストパーティを偽装する最新のトラッカーには効果を発揮しません。

### `$match-case` （大文字・小文字の厳密な区別）
* **構文例**： `||example.com/AdBanner^$match-case`
* **概要**： URLの文字列について、大文字と小文字を厳密に区別して照合します。
* **解説**： 大文字・小文字が意味を持つ環境で正規リソースを巻き込まずに広告のみをピンポイントで遮断できる**メリット**があります。しかし、配信側がURLを小文字に変更するだけで容易にすり抜けられるため、ルールの保守性が低下する**デメリット**があります。

### `$script`, `$image`, `$xmlhttprequest` 等 （リソースタイプの指定）
* **構文例**： `||example.com/api/^$xmlhttprequest`
* **概要**： 通信の種類（スクリプト、画像、XHR/Fetchなど）を限定してルールを適用します。
* **解説**： ブロック対象を正確に絞り込むことで、サイトの必須リソースの巻き込み誤爆を防ぎます。不具合の報告・修正において、「どの種類の通信が遮断されたことでサイトが壊れたのか」を正確にコントロールするために不可欠な修飾子です。

### `$ping` / `$beacon` （トラッキング用ビーコンの指定）
* **構文例**： `||example.com/track^$ping`
* **概要**： `navigator.sendBeacon()` 等によるPing/ビーコンリクエストに限定してルールを適用します。
* **解説**： ページ離脱時などにバックグラウンドで送信されるユーザーの行動トラッキング通信を、通常の通信と区別して確実に遮断するために使用されます。

### `$popup` （ポップアップの遮断）
* **構文例**： `||example.com^$popup`
* **概要**： 対象URLが新しいタブやウィンドウ（ポップアップ）として開かれる動作を検知してブロックします。
* **解説**： ユーザーの意図しない迷惑なリダイレクトやポップアップ広告を確実に防げる**メリット**がありますが、サイトの正規の別窓リンク（ログイン画面や外部決済など）まで巻き込んで遮断してしまう誤爆リスクが常に伴う**デメリット**があります。

### 動作の変更・通信の書き換え
### `$empty` （空のレスポンスの返却）
* **構文例**： `||example.com/tracker.js$empty`
* **概要**： 通信を遮断する際、ブラウザに接続エラーを返すのではなくダミーの「空のレスポンス」を返却します。
* **解説**： スクリプトエラーを検知するアンチアドブロックを回避してサイトの挙動崩れを防げる**メリット**がある一方で、通信自体は成功した（200 OK等）扱いになるため、ネットワークパネルでのデバッグ時に本当にブロックされたのか直感的に把握しづらくなる**デメリット**があります。

### `$redirect` / `$redirect-rule` （ローカルリソースへの置換）
* **構文例**： `||example.com/analytics.js$redirect=noopjs`
* **概要**： 外部スクリプトへのリクエストを遮断し、AdGuardが内包する無害なダミーリソース（空のJSなど）にすり替えます。
* **解説**： スクリプトの読み込み自体が必須要件となっているサイトでの「無限ロード」や「エラーによるレンダリング停止」を回避できます。しかし、MV3環境下では拡張機能がリダイレクトできるリソースに厳密な制限があるため、環境によっては意図通りに動作しない場合があります。

### `$removeparam` （トラッキングパラメータの高度な削除）
* **構文例**： `||example.com^$removeparam=/^utm_[a-z]+=/`
* **概要**： 正規表現を用いて、`utm_` から始まるすべてのクエリパラメータを動的に削除します。
* **解説**： 通信自体は維持されるため、ページ遷移やリダイレクトを破壊せずにプライバシー保護を透過的に実行できる一方、サイトのルーティングやセッション維持に必須のパラメータまで巻き込んで削除してしまうリスクがあります。

### `$replace` （レスポンスボディの直接書き換え）
* **構文例**： `||example.com/api/$replace=/ad_enabled":true/ad_enabled":false/`
* **概要**： 正規表現を用いて、テキストベースのレスポンス（HTMLやJSONなど）の中身を通信経路上で動的に置換します。
* **解説**： DOM操作の前に変数を書き換えられるため極めて強力かつ検知されにくい**メリット**があります。反面、AdGuard for Windows/Mac/Androidなどのシステムレベル（MITM）でのみ動作し、ブラウザ拡張機能（MV3含む）では利用できない強い環境依存性がある点が**デメリット**です。

### `$csp` （Content-Security-Policyの強制適用）
* **構文例**： `||example.com^$csp=script-src 'self'`
* **概要**： HTTPレスポンスヘッダーにCSPを強制的に注入し、サードパーティスクリプトなどの実行をブラウザレベルで禁止します。
* **解説**： DOMに到達する前に悪意あるスクリプトをブラウザの根元から強力に封じ込める**メリット**があります。一方で、サイト本来のCSPと競合して正規のスクリプトまで機能不全に陥るリスクが高く、一般ユーザーには原因特定（コンソールのCSPエラー確認）が難しいという**デメリット**を持ちます。

### デバッグ・優先度の制御
### `$stealth` （ステルスモードの無効化）
* **構文例**： `@@||example.com^$stealth`
* **概要**： AdGuardの「ステルスモード（トラッキング保護機能）」による高度な介入（Cookieブロック、リファラー隠蔽、URLパラメータの自動削除など）を特定のサイトで無効化します。
* **解説**： ログインできない、ページ遷移がおかしいといった不具合が、通常の通信ブロックではなくプライバシー保護機能の過剰介入によるものかを見極めるデバッグに必須の修飾子です。原因がステルスモードにある場合、通常のフィルタ修正ではなくステルスモード専用の例外追加が必要になります。

### `$badfilter` （既存ネットワークルールの無効化）
* **構文例**： `||example.com/ads.js$badfilter`
* **概要**： 購読しているベースフィルタ（Easylistなど）に含まれる特定のネットワークルールを無効化します。
* **解説**： 誤爆の原因となっているサードパーティのルールをピンポイントで解除し、サイトの破損を修正できる非常に実用的な記法です。不具合報告においても「どのルールが悪さをしているか」を特定・提示する際に役立ちます。ただし、大元のルールが更新・削除された場合、このルール自体が不要なゴミとして残ってしまうため、定期的な見直しが求められます。※ 非表示ルールの無効化には使用せず、別途例外ルール `#@#` 等を用います。

### `$generichide` / `$genericblock` （汎用ルールの無効化）
* **構文例**： `@@||example.com^$generichide`
* **概要**： 対象サイト上で、特定のドメインに縛られない汎用的なコスメティックルール `$generichide` やネットワークルール `$genericblock` をすべて無効化します。
* **解説**： サイトの表示崩れや通信エラーが起きた際、原因が「汎用ルールの巻き込み誤爆」か「特定サイト向けルールの誤爆」かを切り分けるデバッグ作業で極めて有効です。ただし、適用すると無害な広告まで大量に再表示されるため、原因特定後のピンポイントな修正ルールの作成が前提となります。

### `$important` （絶対的な優先適用）
* **構文例**： `||example.com/ads.js$important`
* **概要**： `@@` から始まる例外ルール（ホワイトリスト）を含む、他のすべてのルールよりも優先してこのブロックルールを強制適用します。
* **解説**： 汎用的な例外ルールをすり抜けてくる悪質なトラッカーや広告を確実に遮断するための強力な手段です。反面、ルールの優先度階層を破壊するため、乱用すると意図しないブロックが発生した際のデバッグが極めて困難になります。

### `@@...$important` （絶対的な例外・ホワイトリスト化）
* **構文例**： `@@||example.com/ads.js$important`
* **概要**： 他のあらゆるブロックルール（`$important` 修飾子を持つ強力なブロックルールを含む）の干渉を無効化し、指定した通信をエンジンの最優先処理として許可（ホワイトリスト化）します。
* **解説**： AdGuardの処理エンジンの仕様として、通常の例外ルール `@@` は `$important` が付与されたブロックルールに打ち消されてしまいます。しかし、例外ルール側にも `$important` を付与することで優先順位を最上位に引き上げ、いかなる遮断ルールからも対象リソースを確実に保護します。サイトの根幹を成す必須APIやスクリプトが、サードパーティ製フィルタの強力な汎用ルールによって誤爆・遮断されている場合、それを確実に救済・復旧できる**メリット**があります。一方で、ルールとしての権力が強すぎるため、パスの指定やドメインの絞り込みが甘いと、本来防ぐべき悪質なトラッカーや広告まで無条件に通過させてしまう重大な**デメリット**を持ちます。利用する際は `$script` や `$domain=` 等と併用し、適用範囲を極限まで絞り込む緻密な設計が求められます。

## 2. コスメティックルール（DOM要素の非表示）

AdGuard for Androidでも軽く動作させるため、可能な限りシンプルな標準CSSセレクタ `##` を最優先します。

### 基本的な非表示・除外
### `##` （基本のCSS非表示）
* **構文例**： `example.com##.ad-container`
* **概要**： ブラウザネイティブのレンダリングエンジンで直接処理される、最も基本的な要素の非表示記法です。
* **解説**： DOM操作の中で最も処理遅延とバッテリー消費が少ないため最優先で利用すべき記法ですが、クラス名やIDがランダム生成されるモダンな広告要素の特定には無力であるという弱点があります。

### `~` （特定のドメインや要素の適用除外）
* **構文例**： `example.com,~shop.example.com##.ad-banner`
* **概要**： ネットワークルールだけでなく、非表示ルールの適用対象からも特定のサブドメインやクラスを除外します。非表示ルールの例外化には `#@#` も併用されます。
* **解説**： 親ドメイン全体にルールを適用しつつ、表示崩れが起きる特定のサブドメインのみを容易にホワイトリスト化できる**メリット**があります。ただし、除外対象が長くなると可読性が下がり、ルール管理が複雑化する**デメリット**があります。

### 拡張セレクタによる高度な要素指定
### `#?#` （拡張CSSルールの明示的適用）
* **構文例**： `example.com#?#.ad-box:contains(スポンサー)`
* **概要**： 標準のCSSでは指定できない要素を、AdGuard独自の拡張セレクタ（Extended CSS）を用いて指定・非表示にします。
* **解説**： `:contains` や `:has` などを用いた複雑な要素指定を行う際の基本構文です。構造が難読化された広告枠を特定できる強力なメリットがある一方で、ネイティブのCSSエンジンではなくJavaScriptを介してDOMツリーを評価するため、多用するとページのレンダリング遅延やバッテリー消費を増大させるデメリットがあります。AdGuardの仕様上 `##` の後に記述しても自動判別されることがありますが、明示的に `#?#` を用いることで意図しないパースエラーを確実に防ぐことができます。

### `:has` （モダン疑似クラスの活用）
* **構文例**： `example.com##.article-card:has(> .sponsored-label)`
* **概要**： 特定の子要素を持つ親要素をターゲットにします。
* **解説**： 最新のAdGuardエンジンではブラウザのネイティブ実装CSS `has()` に寄せて処理負荷を抑えつつ複雑な構造を非表示にできますが、ネイティブ非対応の古いブラウザ環境ではJavaScriptによるフォールバック評価が行われるためパフォーマンスが著しく低下します。

### `:contains` （テキストフックによる要素指定）
* **構文例**： `example.com#?#.sponsor:contains(PR)`
* **概要**： 指定文字列を含む要素を非表示にします。uBlock Originにおける `:has-text(...)` は使用せず、AdGuard専用の `:contains(...)` を用います。
* **解説**： 構造が難読化された広告枠でも視覚的なテキスト情報を手がかりに確実に消去できます。しかし、DOMツリー全体のテキストノードを走査するため、巨大なページではスクロール時の描画遅延（ジャンク）の原因になり得ます。

### `:upward()` （先祖要素への遡り）
* **構文例**： `example.com#?#.ad-text:upward(.ad-container)`
* **概要**： 引数に指定したCSSセレクター、または数値分だけ階層を遡った先祖要素を探して非表示にします。
* **解説**： 広告のテキストや目印となる子要素を起点にして、ランダムなIDを持つ親コンテナ全体をターゲットにする際に有効です。`:has()` と並び、複雑な構造を持つ広告枠を消去する上で強力な選択肢となります。

### `:xpath()` （XPathを用いた要素指定）
* **構文例**： `example.com#?#:xpath(//div[@id="bar"]/p)`
* **概要**： CSSセレクターの代わりにXPath式を用いて対象要素を探索し、非表示にします。
* **解説**： 親要素の参照だけでなく、特定の兄弟要素の配置関係など、CSSセレクターでは表現できない複雑な条件を持つ要素を指定する最終手段です。ただし、DOM評価の計算コストが高いため、他の記法で代替できない場合のみに限定すべきです。

### `#$?#` （正規表現による動的クラス対応）
* **構文例**： `example.com#$?#div[class^="Ad__"]`
* **概要**： セレクタそのものに正規表現による柔軟なマッチングを適用します。
* **解説**： 難読化によって頻繁に文字列が変化するスクリプト生成型の広告枠に追従できる強力な手法ですが、スタイル再計算の負荷が極めて高くバッテリー消費に直結するため乱用は厳禁です。

### 要素に対するアクション（適用・削除）
### `:style(...)` （CSSプロパティの安全な上書き）
* **構文例**： `example.com##body:style(overflow: auto !important;)`
* **概要**： 要素を隠すのではなく、指定したCSSプロパティを要素に直接適用します。
* **解説**： `#$#` （強制注入）よりも局所的に動作するため、スクロールロック等のアンチアドブロックを安全に解除でき、他への副作用を抑えられる**メリット**があります。しかし、指定した特定の要素にしか影響を与えられないため、ページ全体に関わる複雑な動的スタイル変更には不向きであるという**デメリット**があります。

### `:remove()` （DOMからの物理削除）
* **構文例**： `example.com##.floating-video:remove()`
* **概要**： 単なる `display: none` と異なり、DOMツリーから要素を完全に消去します。
* **解説**： 動画広告などのバックグラウンドでのメモリ消費とCPUリソースの浪費を根本から断ち切ることができます。ただし、一度削除するとページをリロードしない限り復元できないため、SPA（Single Page Application）で誤爆するとサイト全体が機能不全に陥ります。

## 3. 高度なルール（スクリプトレット・CSSインジェクション）

標準機能では突破できないアンチアドブロックに対する最終手段です。uBlock Origin特有の記法 `##+js(...)` などはAdGuardでは解釈されないため、必ずAdGuard固有の構文を使用します。

### `$$` （HTMLフィルタリングルール）
* **構文例**： `example.com$$script[tag-content="ad-script-content"]`
* **概要**： ブラウザがDOMツリーを構築する前の、生HTML応答データの段階で対象の要素を直接物理的に削除します。uBlock Originの `##^` に相当するAdGuard専用の記法です。
* **解説**： DOM読み込みと同時に実行されてしまう悪質なインラインスクリプトやアンチアドブロックを、パースされる前に根本から消去できる極めて強力な機能です。指定できるセレクタは限定的ですが、`tag-content` 擬似属性などを用いてタグ内の文字列を直接検閲できます。

### `#$#` （CSSの強制注入）
* **構文例**： `example.com#$#body { overflow: visible !important; }`
* **概要**： 任意のCSSプロパティをページに直接注入します。
* **解説**： コンテンツブロッカーを検知して `overflow: hidden` でスクロールをロックするような悪質なオーバーレイの制限を強制解除できる一方、サイト自体の正規のレイアウト更新と競合し、予期せぬ表示崩れ（Z-indexの破綻など）を招くリスクが常に伴います。

### `#%#//scriptlet` （スクリプトレットによるJS関数ジャック）
* **構文例**： `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**： ページ読み込みの最初期にAdGuardの組み込みスクリプトを注入し、変数の固定化や特定API `window.open` 等の呼び出しを無効化します。
* **解説**： ネットワークブロックやDOM非表示では回避不可能な高度なJSベースのアンチアドブロックや強制ポップアップを完全に無力化できます。しかし、MV3環境での動的注入制限により将来的に機能しなくなる脆弱性や、対象サイトの僅かなコード変更で即座にルールが陳腐化する欠点があります。

### `#@%#` / `#@?#` （拡張ルールのピンポイント例外）
* **構文例**： `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**： 当該スクリプトレットの注入などを局所的に無効化し、動作を復旧させます。
* **解説**： 購読している強力なベースフィルタが原因でサイトの正規の処理が止まってしまった場合の有効な解決策ですが、どのスクリプトレットが原因で破損しているかを特定するためにはブラウザの開発者ツールを用いた高度なデバッグスキルが要求されます。

## 4. 参考・引用元ソース（最新動向とリポジトリ）

* **[独自の広告フィルタを作成する方法 | AdGuard Knowledge Base](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)**

  AdGuard特有の文法や推奨作法を学べる公式リファレンスです。uBlock Originとの構文の違い `##+js(...)` ではなく `#%#//scriptlet(...)`、`:has-text(...)` ではなく `:contains(...)` を用いる点や、拡張CSSルール `#?#` 等を体系的に習得できるメリットがあります。一方、最新のアンチアドブロック突破などの実践的ロジックは網羅されていないため、uAssets等との併用が必要となる点がデメリットです。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)**

  AdGuard公式の最新リポジトリであり、Android環境へのシステムレベル対応やMV3向け最適化など、AdGuard固有の実践的なコードを直接参照できる最大の強みがあります。ルール数が膨大であるため、目的の記述を探し出すにはリポジトリ構造の深い理解が求められます。

* **[AdGuard Blog](https://adguard.com/en/blog/index.html)**

  広告ブロックの最新トレンドやMV3対応ロードマップ等を発信する公式ブログです。疑似クラス `:has(...)` 等のネイティブ実装状況など、処理負荷を低減する最新のベストプラクティスをいち早く追えるメリットがあります。ただし、一般向けのキャンペーン記事も多く混在しており、フィルタ開発に直結する専門情報を抽出するにはスクリーニングの手間がかかる点がデメリットです。

* **[easylist / easylist](https://github.com/easylist/easylist)**

  すべての原点である汎用ネットワークルールの集積であり、最も枯れた安定したルールの記述法を学べるリポジトリです。MV3環境や最新の難読化広告にはこれ単体では対応しきれない側面もあります。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)**

  高度な動的要素指定やアンチアドブロック突破のロジック群がまとめられており、現代最高峰のフィルタリング技術の参考として有用です。AdGuard環境へ適用するには構文変換として `##+js(...)` を `#%#//scriptlet(...)` に書き換える工程が必須となります。
