## 1. ネットワークルール（通信ブロック）

通信そのものを根元から遮断します。システムへの負荷が最も低いため、あらゆるブロックの基本（第一選択）となります。2026年現在、GitHub上の主要フィルタでも、いかにネットワークルールをシンプルに保ちDNR上限を圧迫しないかが重要視されています。

### 基本構文

* **ブロックの基本**
  * **構文例**: `||example.com^`
  * **解説**: `||`でドメインの先頭を指定し、`^`で区切り文字を指定します。
  * **メリット**: `example.com` およびそのすべてのサブドメインに対する通信を一括で確実に遮断できます。

* **例外（許可）の基本**
  * **構文例**: `@@||example.com^`
  * **解説**: 先頭に `@@` を付与することでブロック対象から除外します。
  * **メリット**: ブロックによってサイトの正常な機能が損なわれた場合、特定の通信のみを許可して動作を修正できます。

### 必須修飾子（Modifiers）一覧

ルールの末尾に `$` を付け、カンマ `,` 区切りで指定します。公式フィルタでも多用される、中級者として必須の修飾子を含みます。

* **適用先の限定（$domain=）**
  * **構文例**: `||example.com/ads.js$domain=test.com|sample.com`
  * **解説**: 指定したドメイン上で読み込まれる場合のみ適用します。`~`を付けることで除外（例: `domain=~test.com`）も可能です。
  * **メリット**: 他のサイトでの正常な通信を巻き込んでブロックしてしまう「誤爆（False Positive）」を防ぎます。

* **外部通信の狙い撃ち（$third-party）**
  * **構文例**: `||tracker.com^$third-party`
  * **解説**: ユーザーが現在訪問しているサイト（ファーストパーティ）とは異なる、外部ドメイン（サードパーティ）への通信に限定してブロックを適用します。
  * **メリット**: サイト自身の動作に必要な内部通信を維持しつつ、外部トラッカーを効率よく狙い撃ちできます。

* **【中級】トラッキングパラメータの削除（$removeparam）**
  * **構文例**: `||example.com^$removeparam=utm_source`
  * **解説**: URLから特定のクエリパラメータ（この場合は`utm_source`）のみを削除し、通信自体は許可します。
  * **メリット**: ページの正常な表示や通信を維持したまま、プライバシーを侵害するトラッキングIDだけをクリーンアップできます。AdGuardの「URLトラッキング防止」フィルタで最も多用されます。

* **【中級】アプリ単位の適用限定（$app） ※Android特化**
  * **構文例**: `||ads.example.com^$app=com.twitter.android`
  * **解説**: 通信ブロックの適用範囲を、指定したAndroidアプリ内のみに限定します。
  * **メリット**: AdGuard for Android環境において、特定のアプリでのみ広告通信を遮断し、他のアプリへの影響を排除できます。

* **【中級】既存ルールの打ち消し（$badfilter）**
  * **構文例**: `||example.com/ad.js$badfilter`
  * **解説**: 購読している公式フィルタなどに含まれる誤爆ルールを無効化します。対象となる元のルールと完全に一致する文字列の末尾に `$badfilter` を追加して記述します。
  * **メリット**: フィルタ更新を待つことなく、サイトの表示崩れを自分自身の手で即座に修正できます。

* **強制ブロック（$important）**
  * **構文例**: `||bad-domain.com^$important`
  * **解説**: 他のいかなる例外ルール（`@@`）よりも優先して、強制的にブロックを適用します。
  * **メリット**: 購読フィルタの例外指定によって悪質な広告がすり抜けてしまう場合に、確実に上書きして遮断できます。

* **サイト破損の防止（$empty）**
  * **構文例**: `||example.com/ad.js$empty`
  * **解説**: 通信を遮断した上で、ブラウザに対して「空のデータ（空のレスポンス）」を返します。
  * **メリット**: 単に通信をタイムアウトさせるよりも、サイト側のスクリプトエラーや無限ロードを防ぐことができ、パフォーマンス低下を回避できます。

## 2. コスメティックルール（DOM要素の非表示）

画面上の見栄えを調整します。Android環境でのレンダリング負荷を下げるため、ブラウザネイティブの標準CSS（`##`）を最優先で使用します。コスメティックルールはDNR上限とは別の枠組みで処理されるため、ネットワークルールの不足を補う重要な役割を担います。

* **基本のCSS非表示**
  * **構文例**: `example.com##.ad-container`
  * **解説**: サイト上の特定のHTML要素を、標準のCSSセレクタを用いて非表示にします。
  * **メリット**: ブラウザのネイティブなレンダリングエンジンで処理されるため、描画遅延やバッテリー消費が極めて少なく最も軽量に動作します。

* **テキスト検索による非表示**
  * **構文例**: `example.com##.sponsor:contains(PR)`
  * **解説**: 指定した文字列（`PR`）を含む要素を検索して非表示にします。uBlock Origin用の `:has-text(...)` は使用せず、AdGuard専用の `:contains(...)` を使用します。
  * **メリット**: クラス名やIDがランダムに自動生成される広告枠であっても、テキストをフックにして確実に取り除くことができます。

* **【中級】コスメティックルールの例外（#@#）**
  * **構文例**: `example.com#@#.ad-container`
  * **解説**: 他のフィルタリストで非表示にされている特定の要素を、指定したドメイン上で再表示（許可）します。
  * **メリット**: サイトの機能に必要な要素が誤って消されている場合に、それをピンポイントで復元できます。

* **【中級】DOMからの完全削除（:remove()）**
  * **構文例**: `example.com##.ad-container:remove()`
  * **解説**: 要素を `display: none` で単に見えなくするのではなく、DOMツリーから物理的に削除します。
  * **メリット**: 無限スクロールなどで大量のダミー要素を生成する重いサイトにおいて、要素自体を消去することでブラウザのメモリ消費やレンダリング負荷を大幅に軽減できます。

## 3. 高度なルール（DOM操作・スクリプトレット）

アンチアドブロック（広告ブロック検知）の回避や、複雑なサイト構造に対応するための最終手段です。処理負荷が高いため、必要最小限の利用に留めます。

* **拡張CSS（疑似クラスの利用）**
  * **構文例**: `example.com#?#.article:-abp-has(.ad)`
  * **解説**: `#?#` を用いてAdGuard独自の拡張CSSを適用し、親要素を遡って指定する場合などに使用します。2026年現在のブラウザ環境では、`:has` などの疑似クラスはほぼ全てのモダンブラウザでネイティブ実装されているため、AdGuardエンジンもそれを利用して処理負荷を大幅に下げる仕様となっています。
  * **メリット**: 標準のCSSセレクタでは到達できない複雑なDOM構造を持つ広告や、ダミー要素でカモフラージュされた広告枠を正確に指定して消去できます。

* **CSSインジェクション（スタイルの強制上書き）**
  * **構文例**: `example.com#$#body { overflow: visible !important; }`
  * **解説**: `#$#` を用いて、要素を非表示にするだけでなく、任意のCSSプロパティをページに直接注入してスタイルを上書きします。
  * **メリット**: 広告ブロッカーを検知して画面のスクロールを固定するような、悪質なオーバーレイやポップアップによる制限を強制的に解除し、正常なブラウジングを復元できます。

* **スクリプトレット（JavaScriptの注入）**
  * **構文例**: `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
  * **解説**: `#%#//scriptlet(...)` を用い、ページ読み込み時にAdGuardが用意した特定のJavaScript関数を実行し、変数を書き換えたり関数の実行を阻止したりします。uBlock Origin用の `##+js(...)` はAdGuardでは機能しません。MV3環境下ではスクリプトの動的注入に厳しい制限があるため多用は避け、Android版との互換性を意識する必要があります。
  * **メリット**: DOMの非表示や通信ブロックだけでは突破できない、高度なアンチアドブロックを無効化できます。

* **【中級】スクリプトレット・拡張CSSの例外（#@%# / #@?#）**
  * **構文例**: `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
  * **解説**: 既存のフィルタリストで注入されている特定のスクリプトレットや拡張CSSを、指定したドメイン上で無効化します。
  * **メリット**: 強力なアンチアドブロック回避ルールが原因で、逆にサイトの正規スクリプトが停止してしまった場合のトラブルシューティングに不可欠です。

### 【発展】実際のコードから設計を読み解く・最新動向を追う

広告ブロック技術の進化の歴史に沿ってソースを辿ることで、ルールの設計思想とその変遷を深く理解できます。

* **[easylist / easylist](https://github.com/easylist/easylist)** （GitHub）
  * **【黎明期〜標準化】** 広告ブロックのデファクトスタンダードであり、すべての原点です。Adblock Plus時代から続く汎用的なネットワークルールが、現在のDNR上限に合わせてどのように統合・最適化されているか、その基礎と設計思想を学べます。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)** （GitHub）
  * **【発展期〜高度化】** 軽量かつ強力なブロックツールとして台頭したuBlock Originの公式フィルタ。記法は異なりますが、動的な要素指定やアンチアドブロック突破のロジックなど、今日の高度なフィルタリング技術の最高峰の参考になります。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)** （GitHub）
  * **【独自進化〜現代】** AdGuard公式フィルタのリポジトリ。システムレベルのブロック（Android等）への対応や、MV3時代に最適化された高度なスクリプトレットの回避策など、現在進行形の実例の宝庫です。

* **[AdGuard Blog](https://adguard.com/en/blog/index.html)**
  * **【最新動向〜未来】** フィルタリングエンジンの更新や、DNR制限に対するAdGuardの最新の回避手法の動向など、これからのルール作成に必須となる最新情報をキャッチアップできます。
