## 1. プラットフォームの特性比較：MV3拡張機能 vs Android

カスタムルールを作成する前に、実行環境における技術的な制限と強みを正しく理解することが不可欠です。

| 機能・特性 | AdGuard ブラウザ拡張機能 (MV3) | AdGuard for Android (CoreLibs) |
| :--- | :--- | :--- |
| **主な動作レイヤー** | ブラウザ内部（DNR API依存） | OSシステム全体（ローカルVPN / MITMプロキシ） |
| **通信遮断の仕組み** | 事前宣言されたDNRルールによるブラウザエンジン処理 | ローカルプロキシでのリクエスト解析・遮断 |
| **ルールの件数上限** | **厳格なDNR制限あり**（通常3万〜最大30万ルール） | **制限なし**（ただし多すぎるとメモリ・バッテリーを消費） |
| **アプリ限定 (`$app`)** | **無効**（ブラウザ内の通信のみ対象のため） | **極めて重要**（対象アプリの物理的隔離） |
| **レスポンス置換 (`$replace`)** | **利用不可**（MV3の権限制限による） | **利用可能**（HTTPSフィルタリング有効時、経路上でテキスト直接改変） |
| **アドバンスド設定** | 基本的なユーザーインターフェースのみ | **ローレベル設定搭載**（パケット・ログ・VPN制御） |
| **最適化の最優先課題** | DNR上限の節約とブラウザレンダリング速度 | バッテリー消費の低減とパケット処理遅延の極小化 |

## 2. 学習手順：基礎からアドバンスド最適化へ導く6ステップ

### 【入門編：IT初心者向け】

### Step 1：アシスタントツールでの体験とプレーンな基本構文の習得

最初はゼロから手書きせず、AdGuard拡張機能の「要素をブロック」機能や、Androidアプリ内の設定を利用して、広告要素がどのようなルールに変換されるかを実体験します。

ここで自動生成される**「プレーンなホスト名指定（`||example.com^`）」**と**「標準的なCSSセレクタ（`##.ad-box`）」**が、すべての環境で最も軽く動作するAdGuardルールの基本構造となります [^1]。

### Step 2：通信遮断と要素隠蔽の使い分け（環境別の意識）

広告の「通信そのものを遮断するルール」と「画面に残った空白を隠すルール」の2種類の役割をマスターします。

特にAdGuard for Androidでは、通信遮断を徹底することでモバイルデータ通信量とバッテリー消費を劇的に削減できます。一方、拡張機能（MV3）では通信遮断がDNRの枠を消費するため、不要に広範なルールを作らないバランス感覚が求められます。

### Step 3：既存の汎用フィルタとの重複回避と例外指定

EasyListやAdGuard Base Filter等の購読フィルタと同じルールを、自分のユーザールールに二重に書かない習慣をつけます [^2]。

自分のカスタムルールは「特定のサイト・アプリでの漏れを補うルール」や「不具合を解除するための例外ルール（`@@`）」など、差分のみをシンプルに管理することが保守性を高める基本です [^4]。

---

### 【応用編：IT中級者向け】

### Step 4：公式・コミュニティリポジトリの査読とAdGuard構文への翻訳

世界中のメンテナーが改善を重ねるリポジトリ（AdGuardFilters, EasyList, uAssets等）のソースコードやIssueを読み解きます。

特にuBlock Origin公式（uAssets）で使われているアンチアドブロック突破の知見を参考にしつつ、そのままコピペするのではなく、AdGuardエンジンに適合した**スクリプトレット（`#%#//scriptlet`）構文**に正しく翻訳して適用するスキルを磨きます [^7]。

### Step 5：フィルタリングログを用いた切り分けと応急処置

ブラウザの開発者ツールと、AdGuard拡張機能・Androidアプリ内蔵の「フィルタリングログ」を使いこなすデバッグ技術を習得します [^5]。

正規サイトが表示崩れを起こした際（誤爆）、通信ブロックか、DOM非表示か、あるいはAdGuard独自のステルスモードの干渉かを切り分け、`$generichide` や例外ルールを用いて的確に応急処置（Workaround）を記述します [^3]。

### Step 6：MV3のDNR制限とAndroidローレベル設定を見据えた極限最適化

拡張機能版（MV3）の厳しいルール上限（DNR）をすり抜けるため、無駄な正規表現を廃し、特異度を調整するモダン疑似クラス（`:is()`, `:where()`）を駆使してコード量を削減します [^6][^8]。

さらに、Android環境においては次章で解説する「アドバンスド設定（ローレベル設定）」を理解し、ログレベルやパケット解析深度を最適化して、システム遅延の極小化を目指します。

## 3. Android版「アドバンスド設定（ローレベル設定）」の解剖と最適化技法

AdGuard for Androidにおける「アドバンスド設定（詳細設定 / ローレベル設定）」は、OSのネットワークスタックやパケット解析エンジン（CoreLibs）に直接介入するための専門的なパラメータ群です。

**【アクセス手順】**

アプリの `「設定(⚙️)」` → `「一般設定」` → `（高度な設定モードをオン）` → `「詳細設定」` → `「ローレベル設定」` へと進みます。

### 3.1 カスタムルール作成とシステム最適化に役立つ主要設定項目

#### 1. ウォッチドッグ（Watchdog） / VPNの自動復元機能

* **機能概要**：
  Android OSの厳しい省電力機能によってAdGuardのバックグラウンドサービスが終了させられた際、自動的にローカルVPNプロセスを監視し再起動・復元する保護機能です。

* **活用とベストプラクティス**：
  Android端末のシステム設定側でAdGuardのバッテリー使用量を「制限なし（無制限）」に設定した上で本機能を有効化することで、意図しない広告ブロックの停止や、それに伴うカスタムルールの適用不全を防ぎます。

#### 2. DNS-over-HTTPS（DoH）フィルタリングの制御

* **機能概要**：
  端末上のアプリがシステムのDNSを迂回し、外部のDoH（暗号化DNS）サーバーへ向けて送信するリクエストをローカルプロキシでインターセプトし、AdGuardのフィルタリングルールを強制適用させる制御機能です。

* **活用とベストプラクティス**：
  モダンなブラウザやアプリが暗号化DNSを用いて広告トラフィックを隠蔽・すり抜けさせる試みを阻止できます。「DNSプロキシにリダイレクトする（Redirect to DNS proxy）」設定で確実なブロックが実現できる反面、厳格なセキュリティ証明書検知（SSLピニング等）を行う一部の金融・通信アプリで接続エラーを誘発するため、必要に応じて除外設定（`$app=` やアプリの管理設定）を行うことが重要です。

#### 3. ECH（Encrypted ClientHello）ブロックおよびHTTP/3（QUIC）制御

* **機能概要**：
  最新の暗号化プロトコルであるECHやHTTP/3（QUIC）を介した通信を制御します。ECHが有効な場合、接続先ホスト名が暗号化されるため、通常のSNI監視によるドメインブロックが困難になる場合があります。

* **活用とベストプラクティス**：
  ローレベル設定で「ECHブロッキング」を有効化、および「HTTP/3のフィルタリング」や「QUICバイパス」を適切に制御することで、最新のプロトコルで保護された広告配信を無力化し、確実にカスタムルールをヒットさせることができます。

#### 4. IPv6のDNS問い合わせ制御 / AAAAレコードブロック

* **機能概要**：
  ドメイン名に対するIPv6アドレスの問い合わせ（AAAAレコード要求）に対して、アプリが自動的に「空のレスポンス（NXDOMAIN / IPv6アドレスが存在しない旨）」を返すよう強制するオプションです。

* **活用とベストプラクティス**：
  IPv6経路でのトラフィックすり抜けや、モバイル回線の不安定なIPv6ルーティングに起因するパケット詰まり・通信遅延を防止します。デメリットとして、IPv6専用ネットワーク環境や、IPv6接続が必須要件となる一部の最新ネットワークサービスにアクセスできなくなるリスクがあります。

#### 5. ログ出力レベル（Logging Level）とトラフィックエクスポート

* **機能概要**：
  AdGuardアプリが記録するログの詳細度（デフォルト、デバッグ、トレース等）および「システムログとPCAPパケットキャプチャ」のエクスポートを制御します。

* **活用とベストプラクティス**：
  自作したスクリプトレット（`#%#//scriptlet`）や通信改変ルール（`$replace` 等）が対象アプリで機能しない場合、ログレベルを一時的に「デバッグ（Debug）」に引き上げてトラフィックを記録・解析することで、難読化された通信パターンを正確に特定できます。ただし、デバッグログはCPU負荷とストレージ書き込みを大幅に増大させるため、検証終了後は速やかに「デフォルト」に戻すことが鉄則です。

### 3.2 アドバンスド設定（ローレベル設定）調整のメリットとデメリット

* **メリット**：
  OSの通信レイヤーや暗号化ハンドシェイクに深く介入できるため、通常の広告ブロッカーを容易にすり抜ける最新の難読化トラッカー、アンチDPI（ディープ・パケット・インスペクション）通信、アプリ内ビルトイン広告を徹底的に遮断・デバッグできます。

* **デメリット**：
  システムネットワークの底層を変更するため、誤ったパケットタイムアウトやリダイレクトパラメータを指定すると、端末全体のインターネット接続が停止したり、著しいバッテリーの異常消耗、アプリの連続クラッシュを引き起こします。

**【公式メンテナーからの警告】**

Andrey Meshkov氏らAdGuard開発チームは、 **「機能の技術的意味を完全に理解している場合や、公式サポートチームからトラブルシューティングのために明示的な指示があった場合を除き、ローレベル設定はデフォルトのまま触れないことがベストプラクティスである」** と強く強調しています。

## 4. ルール設計の鉄則：ベストプラクティスと環境別戦略

### 【初心者が守るべき3つの鉄則】

### 鉄則1：最初は「強力すぎる修飾子（`$all` や `$important`）」を封印する

* **背景・課題**：
  初心者ほど、広告を完全に消そうとして強制力を持つパラメータを多用しがちです。

* **ベストプラクティス**：
  ドメインの末尾に `$` を付ける際は、まずはタイプを指定しないシンプルな状態か、`$script`, `$image` 等の特定タイプのみに留めます。特にMV3拡張機能では、`$all` などの大雑把なルールは意図せぬDNRルールの浪費や正規通信の巻き込みに直結するため、原則使用禁忌とします。

* **メリット**：
  強力な修飾子を避けることで、サイトの認証システムやログイン機能を巻き込んでページやアプリを破壊するリスクを大幅に減らせます。

* **デメリット**：
  強制力が弱い分、悪質なトラッカーや巧妙な広告がすり抜ける可能性がありますが、それは個別リソースの指定で厳密に補うのが正統派です。

### 鉄則2：「通信ブロック」を主役とし、「要素隠蔽」を脇役にする

* **背景・課題**：
  広告を画面で見えなくすること（要素隠蔽）ばかりに頼ると、バックグラウンドで広告データのダウンロードやトラッキング通信が走り続け、端末のリソースを無駄に消費します。

* **ベストプラクティス**：
  アドサーバー（広告配信元）のドメインが判明している場合は、必ず `||ad-server.com^` のように通信そのものを根元から遮断するルールを記述します。コスメティックルール（`##.ad-banner`）は、通信を止められないファーストパーティ配信の広告枠や、空白の枠線を消すためだけに補助的に使用します。

* **メリット**：
  通信を元から断つことで、特にAndroid環境におけるモバイルデータ通信量とパケット処理負荷が減り、バッテリー持ちが劇的に向上します。

* **デメリット**：
  配信元ホスト名が頻繁に変わるスクリプト生成型の広告には追従できず、都度ホストのログ確認が必要になります。

### 鉄則3：自作フィルタは最小限に保ち、複雑な不具合は公式チームへ報告する

* **背景・課題**：
  個人で数千行に及ぶ巨大なカスタムルールを維持・管理することは困難です。

* **ベストプラクティス**：
  自作のユーザールールは数十〜数百行程度の「自身の利用環境向け調整用」に留めましょう。また、AdGuardエンジンはマッチング高速化のためにURLを4096文字で切り詰める仕様があるため、長すぎるURLをそのままルール化しないよう注意が必要です。汎用フィルタですり抜ける広告を見つけた場合は、GitHubのオープンなIssue等を通じてAdGuard公式メンテナーに報告し、大元のリスト（AdGuardFilters）を修正してもらうのが最も効率的です。

* **メリット**：
  日々のルール更新や仕様変更への追従という重いメンテナンス負荷を公式チームやコミュニティに分散でき、常に最適な状態を保てます。

* **デメリット**：
  報告から公式リストに修正が反映され、自身のアプリや拡張機能に配信されるまで数日間〜数週間のタイムラグが生じます。

---

### 【中級者が極めるべき4つの鉄則】

### 鉄則4：ネットワークルールは「適用先（`$domain=`）」と「アプリ（`$app=`）」で隔離する

* **背景・課題**：
  広範な通信ブロックは、共有CDNを利用する正規サイトや、OS内の他アプリの正常通信までまとめて破壊します。

* **ベストプラクティス**：
  特定のウェブサイトでのみ悪用されているスクリプトには、`||example.com/ads.js$domain=test.com` と発信元を厳密に指定します。AdGuard for Androidでは、アプリ内広告を消すルールには必ず `||ads.example.net^$app=com.example.targetapp` とアプリ識別子を指定し、他アプリの通信への干渉を物理的に排除します。

* **メリット**：
  ピンポイントな制御により、他の優良ウェブサイトや他アプリの正常な動作を一切妨げずに、対象の広告・トラッキングのみを確実に遮断できます。

* **デメリット**：
  サイトやアプリ側がサブドメインや配信ルートを変更した際、ルールのメンテナンス漏れにより突然広告が復活するケースがあります。

### 鉄則5：コスメティックルールは疑似クラスを活用し、「スタイル再計算コスト」を極小化する

* **背景・課題**：
  HTML要素を消去するルールは、ブラウザの描画エンジンが画面を描画するたびに計算されます。

* **ベストプラクティス**：
  標準CSSセレクタ `##` で記述できる要素を最優先します。複雑な条件を組む際は `:is(.ad, .sponsor):where([id*="banner"])` のように、特異度（Specificity）をリセットする `:where()` を活用し、ブラウザのスタイル評価負荷を抑えます [^6]。また、uBlock Originの記法をそのまま使うのではなく、AdGuardのエンジンがネイティブ解釈できる形に落とし込むことが重要です。

* **メリット**：
  スタイル再計算の負荷が軽くなり、特にAndroidのモバイルブラウザ環境や低スペック端末において、ページスクロール時のカクつき（レンダリングジャンク）を防止できます。

* **デメリット**：
  `:matches-css()` や `:matches-attr()`、正規表現セレクタ `#$?#` などの重い独自拡張セレクタは、DOM全体の評価と計算をリアルタイムで行うため、バッテリー消費と遅延が跳ね上がります。これらは他手法で代替できない場合の「最終手段」とすべきです。

### 鉄則6：スクリプトレット注入はAdGuard専用記法（`#%#//scriptlet`）を使用する

* **背景・課題**：
  JavaScript実行レイヤーへの介入は、アンチアドブロックを無力化する極めて強力なアプローチです。

* **ベストプラクティス**：
  uBlock Origin特有の `##+js(...)` をそのまま書くのではなく、AdGuardエンジンの実行効率と互換性に最適化された `#%#//scriptlet(...)` 構文に統一します。検知スクリプトに対しては `abort-on-property-read`、動画広告などのインラインデータには `json-prune` を局所的に適用します [^7]。なお、MV3拡張機能ではブラウザのセキュリティ仕様により一部スクリプトレットが制限されるため、不都合があればAdGuard for Android側（OSレベルブロック）での対策を検討します。

* **メリット**：
  通常の通信・DOMブロックでは突破不可能な最新のアンチアドブロック機能や、JSONデータに不可分で組み込まれた広告を、エラーを発生させずに正確に削ぎ落とせます。

* **デメリット**：
  サイト側のJSコードの変数名やJSONスキーマが少しでも変更されると即座に機能しなくなります。また、強力なスクリプト割り込みにより正常な描画処理まで停止させ、ページがフリーズする重大なリスクを持ちます。

### 鉄則7：誤爆（False Positive）対応は「根本原因の無効化」から行う

* **背景・課題**：
  購読フィルタの影響で正規サイトやAndroidアプリの動作が崩れた際、無関係なルールを闇雲に付け足して解消しようとしてはいけません。

* **ベストプラクティス**：
  AdGuardの「フィルタリングログ」で問題のURLを特定し、その通信をブロックしている特定のルールに対して例外 `@@||example.com^$script` を記述するか、当該ルールのみを無効化する `$badfilter` を適用します [^5]。コスメティックの誤爆であれば、`$generichide` や `$specifichide` を対象ドメインに設定し、汎用ルールと個別ルールのどちらが不具合を起こしているかを論理的に切り分けます [^3]。

* **メリット**：
  既存の購読フィルタ（サードパーティ製リスト）を直接いじらず、自分専用のユーザールールだけで不具合を安全かつ綺麗に解消（Workaround）できます [^4]。

* **デメリット**：
  後に大元のリスト側で公式修正がなされた場合、自分で書いた `$badfilter` や例外ルールが「不要なレガシールール（ゴミ）」として残留するため、定期的な見直しと整理が必要です。

## 5. 専門用語解説と脚注（Reference Glossary）

本レポートで使用されている専門用語の解説と、参照すべき公式・信頼性の高い正確なリファレンスURLです。

* **[^1] アシスタントツール / 要素選択ツール**

  画面上で消したい広告をマウスやタップで選択し、対応するブロックルールを自動生成するツール。AdGuardブラウザ拡張機能やuBlock Originに搭載されており、初学者の学習に最適です。

  * **参照**：[uBlock Origin Wiki - Element picker](https://github.com/gorhill/ublock/wiki/Element-picker) / [AdGuard ナレッジベース - 独自の広告フィルタを作成する方法](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)

* **[^2] 汎用フィルタ（EasyList / AdGuard Base Filter）**

  世界中の一般的な広告やトラッカーを遮断するため、コミュニティやAdGuard公式チームによって日夜メンテナンスされている基準となるルールリスト。

  * **参照**：[EasyList Official](https://easylist.to/) / [AdGuard Filters Repository](https://github.com/AdguardTeam/AdguardFilters)

* **[^3] 誤爆（False Positive）**

  本来広告ではない正規のウェブコンテンツやアプリの必須機能（ログイン、動画再生、決済画面等）が、ブロックルールに誤って合致して機能不全に陥る現象。

  * **参照**：[AdGuard ナレッジベース - AdGuardフィルタ一覧と不具合・誤爆報告について](https://adguard.com/kb/general/ad-filtering/adguard-filters/)

* **[^4] 応急処置（Workaround）**

  フィルタの誤爆やサイト・アプリの破損が発生した際、公式リストで根本修正されるまでの間、ユーザーが独自にユーザールールへ追加する一時的な回避策や例外ルールのこと。

* **[^5] フィルタリングログ（Filtering Log）**

  AdGuardのブラウザ拡張機能やAndroidアプリに内蔵されている監視機能。どの通信が「どのフィルタリストのどのルール」によって遮断・許可されたかをリアルタイムに確認・デバッグできます。

  * **参照**：[AdGuard Blog - In-Depth Features Overview: AdGuard for Android](https://adguard.com/en/blog/in-depth-review-adguard-for-android.html)

* **[^6] 特異度（Specificity）**

  CSSセレクタが「どれだけ詳細に特定の要素を指定しているか」を表す計算値。`:where()` 疑似クラスを使うと特異度を0にリセットでき、スタイルの競合やブラウザの計算負荷を最適化できます。

  * **参照**：[uBlock Origin Wiki - Static filter syntax](https://github.com/gorhill/ublock/wiki/static-filter-syntax)

* **[^7] スクリプトレット（Scriptlet）**

  ページ読み込みの初期段階にAdGuardエンジンが注入する小さなJSプログラム。アンチアドブロックの検知関数をフックして無力化したり、インラインデータの広告を取り除きます。

  * **参照**：[AdGuard ナレッジベース - 独自の広告フィルタを作成する方法](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)

* **[^8] MV3（Manifest V3）とDNR（Declarative Net Request）**

  ブラウザ拡張機能の最新セキュリティ仕様（MV3）。拡張機能が通信を直接改変する権限が制限され、事前に宣言したルール（DNR）に基づきブラウザ自身が高速にブロックする仕様です。ルール数に厳密な上限（3万〜30万）が存在します。

  * **参照**：[AdGuard ナレッジベース - 独自の広告フィルタを作成する方法（DNR互換性解説）](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)

* **[^9] DOM（Document Object Model）**

  ウェブページを構成するHTMLやXMLの要素をツリー状の構造として表現した仕組み。コスメティックルールはこのDOMツリーを走査・評価して広告要素を見つけ出し非表示にします。
