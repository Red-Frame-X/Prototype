## 1. ネットワークルール（通信ブロック）

通信そのものを根元から遮断します。DNR上限を圧迫しないよう、正規表現の乱用を避け、可能な限りプレーンなホスト名指定を行うのが上級者の基本です。

### 基本構文と高度な修飾子（Modifiers）

ルールの末尾に `$` を付け、カンマ `,` 区切りで指定します。

* **適用先の限定と除外（$domain= / $denyallow=）**
  * **構文例**: `||example.com/ads.js$domain=test.com|~sub.test.com`
  * **構文例**: `||tracker.com^$denyallow=allowed.com|trusted.com`
  * **解説**: `$domain` は指定ドメインでのみブロックを有効にし、`~` で特定サブドメインを除外します。`$denyallow` は逆に「指定したドメイン以外」のすべての場所でブロックを適用します。
  * **メリット**: 複雑なサイト群における誤爆（False Positive）を極めて正確にコントロールできます。
  * **デメリット**: サブドメインの運用が変更された際、ルールのメンテナンス漏れによるサイト破損を引き起こしやすいです。

* **トラッキングパラメータの高度な削除（$removeparam）**
  * **構文例**: `||example.com^$removeparam=/^utm_[a-z]+=/`
  * **解説**: 正規表現を用いて、`utm_` から始まるすべてのクエリパラメータを動的に削除します。
  * **メリット**: 通信自体は維持されるため、ページ遷移やリダイレクトを破壊せずにプライバシー保護を透過的に実行できます。
  * **デメリット**: サイトのルーティングやセッション維持に必須のパラメータまで巻き込んで削除してしまうリスクがあります。

* **ローカルリソースへの置換（$redirect / $redirect-rule）**
  * **構文例**: `||example.com/analytics.js$redirect=noopjs`
  * **解説**: 外部スクリプトへのリクエストを遮断し、AdGuardが内包する無害なダミーリソース（空のJSなど）にすり替えます。
  * **メリット**: スクリプトの読み込み自体が必須要件となっているサイトでの「無限ロード」や「エラーによるレンダリング停止」を回避できます。
  * **デメリット**: MV3環境下では拡張機能がリダイレクトできるリソースに厳密な制限があるため、環境によっては意図通りに動作しません。

* **アプリ単位の適用限定（$app）※Android特化**
  * **構文例**: `||ads.example.com^$app=com.twitter.android`
  * **メリット**: システム全体を監視するAdGuard for Android環境において、特定のアプリにのみルールを適用し、他アプリの正常なバックグラウンド通信を妨げません。
  * **デメリット**: デスクトップ版のブラウザ拡張機能では完全に無視されるため、クロスプラットフォームでのルール共有には不向きです。

## 2. コスメティックルール（DOM要素の非表示）

AdGuard for Androidでも軽く動作させるため、可能な限りシンプルな標準CSSセレクタ `##` を最優先します。

* **基本のCSS非表示（##）**
  * **構文例**: `example.com##.ad-container`
  * **メリット**: ブラウザネイティブのレンダリングエンジンで処理されるため、DOM操作の中で最も処理遅延とバッテリー消費が少ないです。
  * **デメリット**: クラス名やIDがランダム生成されるモダンな広告要素の特定には無力です。

* **テキストフックによる要素指定（:contains）**
  * **構文例**: `example.com##.sponsor:contains(PR)`
  * **解説**: 指定文字列を含む要素を非表示にします。uBlock Originにおける `:has-text(...)` は使用せず、AdGuard専用の `:contains(...)` を用います。
  * **メリット**: 構造が難読化された広告枠でも、視覚的なテキスト情報を手がかりに確実に消去できます。
  * **デメリット**: DOMツリー全体のテキストノードを走査するため、巨大なページではスクロール時の描画遅延（ジャンク）の原因になり得ます。

* **モダン疑似クラスの活用（:has）**
  * **構文例**: `example.com##.article-card:has(> .sponsored-label)`
  * **解説**: 特定の子要素を持つ親要素をターゲットにします。
  * **メリット**: 最新のAdGuardエンジンでは、ブラウザのネイティブ実装（CSS `has()`）が存在する場合そちらに寄せて処理されるため、負荷を抑えつつ複雑な構造を非表示にできます。
  * **デメリット**: ネイティブ非対応の古いブラウザ環境では、JavaScriptによるフォールバック評価が行われるため、パフォーマンスが著しく低下します。

* **正規表現による動的クラス対応（#$?#）**
  * **構文例**: `example.com#$?#div[class^="Ad__"]`
  * **解説**: セレクタそのものに正規表現による柔軟なマッチングを適用します。
  * **メリット**: 難読化によって頻繁に文字列が変化するスクリプト生成型の広告枠に追従できます。
  * **デメリット**: スタイル再計算の負荷が極めて高く、バッテリー消費に直結するため、乱用は厳禁です。

* **DOMからの物理削除（:remove()）**
  * **構文例**: `example.com##.floating-video:remove()`
  * **メリット**: 単なる `display: none` と異なり、DOMツリーから要素を完全に消去するため、動画広告などのバックグラウンドでのメモリ消費とCPUリソースの浪費を根本から断ち切ります。
  * **デメリット**: 一度削除するとページをリロードしない限り復元できないため、SPA（Single Page Application）で誤爆するとサイト全体が機能不全に陥ります。

## 3. 高度なルール（スクリプトレット・CSSインジェクション）

標準機能では突破できないアンチアドブロックに対する最終手段です。uBlock Origin特有の記法（`##+js(...)`など）はAdGuardでは解釈されないため、必ずAdGuard固有の構文を使用します。

* **CSSの強制注入（#$#）**
  * **構文例**: `example.com#$#body { overflow: visible !important; }`
  * **解説**: 任意のCSSプロパティをページに直接注入します。
  * **メリット**: 広告ブロッカーを検知して `overflow: hidden` でスクロールをロックするような、悪質なオーバーレイの制限を強制解除できます。
  * **デメリット**: サイト自体の正規のレイアウト更新と競合し、予期せぬ表示崩れ（Z-indexの破綻など）を招くリスクが常に伴います。

* **スクリプトレットによるJS関数ジャック（#%#//scriptlet）**
  * **構文例**: `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
  * **構文例**: `example.com#%#//scriptlet('prevent-window-open')`
  * **解説**: ページ読み込みの最初期にAdGuardの組み込みスクリプトを注入し、変数の固定化や特定API（`window.open`等）の呼び出しを無効化します。
  * **メリット**: ネットワークブロックやDOM非表示では回避不可能な、高度なJavaScriptベースのアンチアドブロックや強制ポップアップを完全に無力化できます。
  * **デメリット**: MV3環境ではスクリプトの動的注入が厳しく制限されているため、将来的なAPIの仕様変更で機能しなくなる脆弱性を孕みます。また、対象サイトの内部ロジックへの強い依存があるため、サイト側の僅かなコード変更で即座にルールが陳腐化します。

* **拡張ルールのピンポイント例外（#@%# / #@?#）**
  * **構文例**: `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
  * **メリット**: 購読している強力なベースフィルタが原因でサイトの正規の処理が止まってしまった場合、当該スクリプトレットの注入のみを局所的に無効化し、動作を復旧できます。
  * **デメリット**: どのスクリプトレットが原因でサイトが破損しているかを特定するためには、ブラウザの開発者ツール（Console等）を用いた高度なデバッグスキルが要求されます。

## 参考・引用元ソース（最新動向とリポジトリ）

* **[easylist / easylist](https://github.com/easylist/easylist)**
  * **概要**: 全ての原点である汎用ネットワークルールの集積。
  * **メリット**: 最も枯れた安定したルールの記述法を学べます。
  * **デメリット**: MV3や最新の難読化広告には単体では対応しきれません。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)**
  * **概要**: 高度な動的要素指定やアンチアドブロック突破のロジック群。
  * **メリット**: 現代の最高峰のフィルタリング技術・設計思想の参考になります。
  * **デメリット**: 記法が異なるため、AdGuard環境へは手動での構文翻訳（`##+js` → `#%#//scriptlet` など）が必須となります。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)**
  * **概要**: AdGuard公式の最新リポジトリ。
  * **メリット**: Android環境へのシステムレベル対応やMV3向け最適化など、AdGuard固有の実践的なコードを直接参照できます。
  * **デメリット**: ルール数が膨大であるため、目的の記述を探し出すにはリポジトリ構造の理解が必要です。

* **[独自の広告フィルタを作成する方法 \| AdGuard Knowledge Base](https://adguard.com/kb/ja/general/ad-filtering/create-own-filters/)**
