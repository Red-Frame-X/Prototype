## 1. ネットワークルール（通信ブロック）

通信そのものを根元から遮断します。DNR上限を圧迫しないよう、正規表現の乱用を避け、可能な限りプレーンなホスト名指定を行うのが上級者の基本です。

### 基本構文と高度な修飾子（Modifiers）

ルールの末尾に `$` を付け、カンマ `,` 区切りで指定します。

* **適用先の限定と除外（$domain= / $denyallow=）**
  * **構文例**： `||example.com/ads.js$domain=test.com|~sub.test.com` または `||tracker.com^$denyallow=allowed.com|trusted.com`
  * **概要**： `$domain` は指定ドメインでのみブロックを有効にし、`~` で特定サブドメインを除外します。`$denyallow` は逆に「指定したドメイン以外」のすべての場所でブロックを適用します。
  * **概要**： 複雑なサイト群における誤爆（False Positive）を極めて正確にコントロールできる強力なメリットがありますが、サブドメインの運用が変更された際、ルールのメンテナンス漏れによるサイト破損を引き起こしやすい点に注意が必要です。

* **トラッキングパラメータの高度な削除（$removeparam）**
  * **構文例**： `||example.com^$removeparam=/^utm_[a-z]+=/`
  * **概要**： 正規表現を用いて、`utm_` から始まるすべてのクエリパラメータを動的に削除します。
  * **概要**： 通信自体は維持されるため、ページ遷移やリダイレクトを破壊せずにプライバシー保護を透過的に実行できる一方、サイトのルーティングやセッション維持に必須のパラメータまで巻き込んで削除してしまうリスクがあります。

* **ローカルリソースへの置換（$redirect / $redirect-rule）**
  * **構文例**： `||example.com/analytics.js$redirect=noopjs`
  * **概要**： 外部スクリプトへのリクエストを遮断し、AdGuardが内包する無害なダミーリソース（空のJSなど）にすり替えます。
  * **概要**： スクリプトの読み込み自体が必須要件となっているサイトでの「無限ロード」や「エラーによるレンダリング停止」を回避できます。しかし、MV3環境下では拡張機能がリダイレクトできるリソースに厳密な制限があるため、環境によっては意図通りに動作しないデメリットがあります。

* **アプリ単位の適用限定（$app）※ Android特化**
  * **構文例**： `||ads.example.com^$app=com.twitter.android`
  * **概要**： システム全体を監視するAdGuard for Android環境において、特定のアプリにのみルールを適用するための指定です。
  * **概要**： 他のアプリの正常なバックグラウンド通信を妨げないという大きな利点がありますが、デスクトップ版のブラウザ拡張機能では完全に無視されるため、クロスプラットフォームでのルール共有には不向きです。

## 2. コスメティックルール（DOM要素の非表示）

AdGuard for Androidでも軽く動作させるため、可能な限りシンプルな標準CSSセレクタ `##` を最優先します。

* **基本のCSS非表示（##）**
  * **構文例**： `example.com##.ad-container`
  * **概要**： ブラウザネイティブのレンダリングエンジンで直接処理される、最も基本的な要素の非表示記法です。
  * **概要**： DOM操作の中で最も処理遅延とバッテリー消費が少ないのが最大のメリットですが、クラス名やIDがランダム生成されるモダンな広告要素の特定には無力であるという弱点があります。

* **テキストフックによる要素指定（:contains）**
  * **構文例**： `example.com##.sponsor:contains(PR)`
  * **概要**： 指定文字列を含む要素を非表示にします。uBlock Originにおける `:has-text(...)` は使用せず、AdGuard専用の `:contains(...)` を用います。
  * **概要**： 構造が難読化された広告枠でも視覚的なテキスト情報を手がかりに確実に消去できますが、DOMツリー全体のテキストノードを走査するため、巨大なページではスクロール時の描画遅延（ジャンク）の原因になり得ます。

* **モダン疑似クラスの活用（:has）**
  * **構文例**： `example.com##.article-card:has(> .sponsored-label)`
  * **解説**： 特定の子要素を持つ親要素をターゲットにします。
  * **概要**： 最新のAdGuardエンジンではブラウザのネイティブ実装（CSS `has()`）に寄せて処理負荷を抑えつつ複雑な構造を非表示にできますが、ネイティブ非対応の古いブラウザ環境ではJavaScriptによるフォールバック評価が行われるためパフォーマンスが著しく低下します。

* **正規表現による動的クラス対応（#$?#）**
  * **構文例**： `example.com#$?#div[class^="Ad__"]`
  * **解説**： セレクタそのものに正規表現による柔軟なマッチングを適用します。
  * **概要**： 難読化によって頻繁に文字列が変化するスクリプト生成型の広告枠に追従できる強力な手法ですが、スタイル再計算の負荷が極めて高くバッテリー消費に直結するため乱用は厳禁です。

* **DOMからの物理削除（:remove()）**
  * **構文例**： `example.com##.floating-video:remove()`
  * **解説**： 単なる `display: none` と異なり、DOMツリーから要素を完全に消去します。
  * **概要**： 動画広告などのバックグラウンドでのメモリ消費とCPUリソースの浪費を根本から断ち切ることができます。ただし、一度削除するとページをリロードしない限り復元できないため、SPA（Single Page Application）で誤爆するとサイト全体が機能不全に陥ります。

## 3. 高度なルール（スクリプトレット・CSSインジェクション）

標準機能では突破できないアンチアドブロックに対する最終手段です。uBlock Origin特有の記法はAdGuardでは解釈されないため、必ずAdGuard固有の構文を使用します。

* **CSSの強制注入（#$#）**
  * **構文例**： `example.com#$#body { overflow: visible !important; }`
  * **解説**： 任意のCSSプロパティをページに直接注入します。
  * **概要**： コンテンツブロッカーを検知して `overflow: hidden` でスクロールをロックするような悪質なオーバーレイの制限を強制解除できる一方、サイト自体の正規のレイアウト更新と競合し、予期せぬ表示崩れ（Z-indexの破綻など）を招くリスクが常に伴います。

* **スクリプトレットによるJS関数ジャック（#%#//scriptlet）**
  * **構文例**： `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
  * **解説**： ページ読み込みの最初期にAdGuardの組み込みスクリプトを注入し、変数の固定化や特定API（`window.open`等）の呼び出しを無効化します。
  * **概要**： ネットワークブロックやDOM非表示では回避不可能な高度なJSベースのアンチアドブロックや強制ポップアップを完全に無力化できます。しかし、MV3環境での動的注入制限により将来的に機能しなくなる脆弱性や、対象サイトの僅かなコード変更で即座にルールが陳腐化する欠点があります。

* **拡張ルールのピンポイント例外（#@%# / #@?#）**
  * **構文例**： `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
  * **解説**： 当該スクリプトレットの注入などを局所的に無効化し、動作を復旧させます。
  * **概要**： 購読している強力なベースフィルタが原因でサイトの正規の処理が止まってしまった場合の有効な解決策ですが、どのスクリプトレットが原因で破損しているかを特定するためにはブラウザの開発者ツールを用いた高度なデバッグスキルが要求されます。

## 参考・引用元ソース（最新動向とリポジトリ）

* **[easylist / easylist](https://github.com/easylist/easylist)**
  * **概要**： 全ての原点である汎用ネットワークルールの集積であり、最も枯れた安定したルールの記述法を学べるという利点があります。その一方で、MV3環境や最新の難読化広告にはこれ単体では対応しきれないという側面を持っています。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)**
  * **概要**： 高度な動的要素指定やアンチアドブロック突破のロジック群がまとめられており、現代の最高峰のフィルタリング技術・設計思想の参考として非常に有用です。ただし、記法が異なるため、AdGuard環境へ適用するには手動での構文翻訳（`##+js` を `#%#//scriptlet` に書き換える等）が必須となる手間があります。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)**
  * **概要**： AdGuard公式の最新リポジトリであり、Android環境へのシステムレベル対応やMV3向け最適化など、AdGuard固有の実践的なコードを直接参照できるのが最大の強みです。しかし、格納されているルール数が膨大であるため、目的の記述を探し出すにはリポジトリ構造の深い理解が求められます。
