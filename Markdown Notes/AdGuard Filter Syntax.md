# AdGuard Filter Syntax

**【要約】**
MV3（Manifest V3）環境の厳格なDNR（Declarative Net Request）ルール上限や、AdGuard for Androidにおける処理遅延・バッテリー消費の極小化を前提とした、上級者向けのカスタムルール・リファレンスです。
通信遮断、DOM操作、スクリプト注入の各アプローチについて、AdGuard専用の記法（`#$#`、`#?#`、`:contains`、`#%#//scriptlet` 等）を用いた最適な運用手法と注意点を整理しています。

## 1. ネットワークルール（通信ブロック）

通信そのものを根元から遮断します。DNR上限を圧迫しないよう、正規表現の乱用を避け、可能な限りプレーンなホスト名指定を行うのが上級者の基本です。ルールの末尾に `$` を付け、カンマ `,` 区切りで修飾子（Modifiers）を指定します。

### `$domain=` / `$denyallow=` （適用先の限定と除外）
* **構文例**： `||example.com/ads.js$domain=test.com|~sub.test.com`
* **概要**： `$domain` は指定ドメインでのみブロックを有効にし、`~` で特定サブドメインを除外します。`$denyallow` は逆に「指定したドメイン以外」のすべての場所でブロックを適用します。
* **解説**：複雑なサイト群における誤爆（False Positive）を極めて正確にコントロールできる強力な機能ですが、サブドメインの運用が変更された際、ルールのメンテナンス漏れによるサイト破損を引き起こしやすい点に注意が必要です。

### `$badfilter` （既存ルールの無効化）
* **構文例**： `||example.com/ads.js$badfilter`
* **概要**：購読しているベースフィルタ（Easylistなど）に含まれる特定のブロックルールを無効化します。
* **解説**：誤爆の原因となっているサードパーティのルールをピンポイントで解除し、サイトの破損を修正できる非常に実用的な記法です。ただし、大元のルールが更新・削除された場合、このルール自体が不要なゴミとして残ってしまうため、定期的な見直しが求められます。

### `$important` （絶対的な優先適用）
* **構文例**： `||example.com/ads.js$important`
* **概要**： `@@` から始まる例外ルール（ホワイトリスト）を含む、他のすべてのルールよりも優先してこのブロックルールを強制適用します。
* **解説**：汎用的な例外ルールをすり抜けてくる悪質なトラッカーや広告を確実に遮断するための強力な手段です。反面、ルールの優先度階層を破壊するため、乱用すると意図しないブロックが発生した際のデバッグが極めて困難になります。

### `$third-party` （サードパーティリクエストの制限）
* **構文例**： `||tracker.com^$third-party`
* **概要**：リクエストの送信先が、現在閲覧しているサイトのドメインと異なる（サードパーティである）場合のみ適用されます。
* **解説**：CDNやファーストパーティとして動作する必須スクリプトを保護しつつ、外部からのトラッキング通信のみを効率よく遮断できます。ただし、CNAMEクローキングなどを用いてファーストパーティを偽装する最新のトラッカーには効果を発揮しません。

### `$removeparam` （トラッキングパラメータの高度な削除）
* **構文例**： `||example.com^$removeparam=/^utm_[a-z]+=/`
* **概要**：正規表現を用いて、`utm_` から始まるすべてのクエリパラメータを動的に削除します。
* **解説**：通信自体は維持されるため、ページ遷移やリダイレクトを破壊せずにプライバシー保護を透過的に実行できる一方、サイトのルーティングやセッション維持に必須のパラメータまで巻き込んで削除してしまうリスクがあります。

### `$redirect` / `$redirect-rule` （ローカルリソースへの置換）
* **構文例**： `||example.com/analytics.js$redirect=noopjs`
* **概要**：外部スクリプトへのリクエストを遮断し、AdGuardが内包する無害なダミーリソース（空のJSなど）にすり替えます。
* **解説**：スクリプトの読み込み自体が必須要件となっているサイトでの「無限ロード」や「エラーによるレンダリング停止」を回避できます。しかし、MV3環境下では拡張機能がリダイレクトできるリソースに厳密な制限があるため、環境によっては意図通りに動作しない場合があります。

### `$app` （アプリ単位の適用限定）※Android特化
* **構文例**： `||ads.example.com^$app=com.twitter.android`
* **概要**：システム全体を監視するAdGuard for Android環境において、特定のアプリにのみルールを適用するための指定です。
* **解説**：他のアプリの正常なバックグラウンド通信を妨げないという大きな利点がありますが、デスクトップ版のブラウザ拡張機能では完全に無視されるため、クロスプラットフォームでのルール共有には不向きです。

## 2. コスメティックルール（DOM要素の非表示）

AdGuard for Androidでも軽く動作させるため、可能な限りシンプルな標準CSSセレクタ `##` を最優先します。

### `##` （基本のCSS非表示）
* **構文例**： `example.com##.ad-container`
* **概要**：ブラウザネイティブのレンダリングエンジンで直接処理される、最も基本的な要素の非表示記法です。
* **解説**：DOM操作の中で最も処理遅延とバッテリー消費が少ないため最優先で利用すべき記法ですが、クラス名やIDがランダム生成されるモダンな広告要素の特定には無力であるという弱点があります。

### `:contains` （テキストフックによる要素指定）
* **構文例**： `example.com##.sponsor:contains(PR)`
* **概要**：指定文字列を含む要素を非表示にします。uBlock Originにおける `:has-text(...)` は使用せず、AdGuard専用の `:contains(...)` を用います。
* **解説**：構造が難読化された広告枠でも視覚的なテキスト情報を手がかりに確実に消去できます。しかし、DOMツリー全体のテキストノードを走査するため、巨大なページではスクロール時の描画遅延（ジャンク）の原因になり得ます。

### `:has` （モダン疑似クラスの活用）
* **構文例**： `example.com##.article-card:has(> .sponsored-label)`
* **概要**：特定の子要素を持つ親要素をターゲットにします。
* **解説**：最新のAdGuardエンジンではブラウザのネイティブ実装（CSS `has()`）に寄せて処理負荷を抑えつつ複雑な構造を非表示にできますが、ネイティブ非対応の古いブラウザ環境ではJavaScriptによるフォールバック評価が行われるためパフォーマンスが著しく低下します。

### `#$?#` （正規表現による動的クラス対応）
* **構文例**： `example.com#$?#div[class^="Ad__"]`
* **概要**：セレクタそのものに正規表現による柔軟なマッチングを適用します。
* **解説**：難読化によって頻繁に文字列が変化するスクリプト生成型の広告枠に追従できる強力な手法ですが、スタイル再計算の負荷が極めて高くバッテリー消費に直結するため乱用は厳禁です。

### `:remove()` （DOMからの物理削除）
* **構文例**： `example.com##.floating-video:remove()`
* **概要**：単なる `display: none` と異なり、DOMツリーから要素を完全に消去します。
* **解説**：動画広告などのバックグラウンドでのメモリ消費とCPUリソースの浪費を根本から断ち切ることができます。ただし、一度削除するとページをリロードしない限り復元できないため、SPA（Single Page Application）で誤爆するとサイト全体が機能不全に陥ります。

## 3. 高度なルール（スクリプトレット・CSSインジェクション）

標準機能では突破できないアンチアドブロックに対する最終手段です。uBlock Origin特有の記法（`##+js(...)` など）はAdGuardでは解釈されないため、必ずAdGuard固有の構文を使用します。

### `#$#` （CSSの強制注入）
* **構文例**： `example.com#$#body { overflow: visible !important; }`
* **概要**：任意のCSSプロパティをページに直接注入します。
* **解説**：コンテンツブロッカーを検知して `overflow: hidden` でスクロールをロックするような悪質なオーバーレイの制限を強制解除できる一方、サイト自体の正規のレイアウト更新と競合し、予期せぬ表示崩れ（Z-indexの破綻など）を招くリスクが常に伴います。

### `#%#//scriptlet` （スクリプトレットによるJS関数ジャック）
* **構文例**： `example.com#%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**：ページ読み込みの最初期にAdGuardの組み込みスクリプトを注入し、変数の固定化や特定API（`window.open`等）の呼び出しを無効化します。
* **解説**：ネットワークブロックやDOM非表示では回避不可能な高度なJSベースのアンチアドブロックや強制ポップアップを完全に無力化できます。しかし、MV3環境での動的注入制限により将来的に機能しなくなる脆弱性や、対象サイトの僅かなコード変更で即座にルールが陳腐化する欠点があります。

### `#@%#` / `#@?#` （拡張ルールのピンポイント例外）
* **構文例**： `example.com#@%#//scriptlet('set-constant', 'adblockDetected', 'false')`
* **概要**：当該スクリプトレットの注入などを局所的に無効化し、動作を復旧させます。
* **解説**：購読している強力なベースフィルタが原因でサイトの正規の処理が止まってしまった場合の有効な解決策ですが、どのスクリプトレットが原因で破損しているかを特定するためにはブラウザの開発者ツールを用いた高度なデバッグスキルが要求されます。

## 4. 参考・引用元ソース（最新動向とリポジトリ）

* **[easylist / easylist](https://github.com/easylist/easylist)**

  すべての原点である汎用ネットワークルールの集積であり、最も枯れた安定したルールの記述法を学べるリポジトリです。MV3環境や最新の難読化広告にはこれ単体では対応しきれない側面もあります。

* **[uBlockOrigin / uAssets](https://github.com/uBlockOrigin/uAssets)**

  高度な動的要素指定やアンチアドブロック突破のロジック群がまとめられており、現代最高峰のフィルタリング技術の参考として有用です。AdGuard環境へ適用するには手動での構文翻訳（`##+js` を `#%#//scriptlet` に書き換える等）が必須となります。

* **[AdguardTeam / AdguardFilters](https://github.com/AdguardTeam/AdguardFilters)**

  AdGuard公式の最新リポジトリであり、Android環境へのシステムレベル対応やMV3向け最適化など、AdGuard固有の実践的なコードを直接参照できる最大の強みがあります。ルール数が膨大であるため、目的の記述を探し出すにはリポジトリ構造の深い理解が求められます。
